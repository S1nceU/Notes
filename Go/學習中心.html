0.3Makefile：把指令變成團隊介面

Makefile 在 Go 專案裡不是建置系統，而是「指令的目錄」：任何人（包括三個月後的你）clone 下來只需要知道 make lint、make test、make build 三件事。這也是之後 CI 的介面——CI 跑的和你本機跑的是同一條指令，才不會有「CI 才會失敗」的謎團。

兩個語法重點：.PHONY 宣告這些 target 不是檔案名稱（否則目錄裡剛好有個叫 test 的檔案時 make 會說「已是最新」什麼都不做）；縮排必須是 tab 不能是空格，這是 make 四十年來的著名地雷。

重點

go test -race -count=1：-race 開啟 data race 偵測（第 2 週的主角），-count=1 停用測試快取，確保每次都真的執行。日常開發可拿掉 -count=1 換速度。

0.4Git 工作流：branch、commit convention 與 tag

單人專案也照團隊規矩走，因為這個 repo 之後就是你的作品集，面試官會看 git history。約定三件事：

  • Branch：main 永遠可建置；每週的工作開在 feat/w01-job-api 這類短命分支，完成後合回。
  • Commit convention：採 Conventional Commits——feat: add job creation endpoint、fix: return 404 for unknown job id、test:、refactor:、docs:、chore:。第一行 50 字內、祈使句、說「為什麼」而不是「做了什麼」的細節放 body。

0.5Docker、kind 與 kubectl：本機的迷你雲

你的服務最終跑在 Kubernetes 上，所以第 0 週就把「本機 K8s」建好。分工是：Docker 負責把程式打包成 image 並在本機跑容器；kind（Kubernetes IN Docker）用 Docker 容器模擬出一整個 K8s cluster；kubectl 是你對任何 cluster 說話的 CLI。

自我測驗 · Week 0

先自己回答，再展開對照。答不出來就回去重讀該節。

Q1go.sum 的作用是什麼？為什麼必須 commit？

顯示答案

它記錄每個相依套件內容的加密雜湊。任何人執行 go mod download 時，Go 會驗證抓到的內容與雜湊相符，確保所有機器使用完全相同的相依內容，防止相依被竄改（供應鏈攻擊）。不 commit 就失去這層保證。

Q2internal/ 目錄和其他目錄有什麼本質差別？

顯示答案

這是編譯器層級的存取控制：internal/ 下的 package 只能被「internal/ 的上一層目錄」以下的程式碼 import，module 外部 import 會編譯失敗。它讓你能公開 repo 卻不必對這些 API 的穩定性負責。

Q3為什麼 CI 應該執行 make test 而不是直接寫 go test ./...？

顯示答案

讓 CI 與本機執行「同一條指令」。當測試指令需要調整（例如加 -race、改 timeout），只改 Makefile 一處，本機與 CI 行為保持一致，不會出現「CI 上跑的參數跟我不一樣」的除錯黑洞。

Q4kind 和 Docker 的關係是什麼？

顯示答案

kind 用 Docker 容器充當 Kubernetes 的「節點」：一個容器裡跑 kubelet、containerd 與 control plane 元件，對 kubectl 而言它就是一個真的 cluster。所以 Docker 壞了 kind 一定也壞，除錯時先確認 docker ps。

面試問答 · Week 0

你的專案為什麼採用 cmd / internal / domain 這樣的結構？

回答框架：(1) cmd 讓一個 repo 能產出多個執行檔（API server、agent、controller），main 只做組裝；(2) internal 用編譯器強制私有邊界；(3) domain / application / transport 分層讓商業邏輯不依賴 HTTP 框架，可以獨立測試，之後從 REST 加上 gRPC 時 domain 層零改動。加一句誠實的收尾：小專案這樣切有前期成本，但這個專案的目標是長到 32 週規模，分層是為了後面付的保險。

go vet、staticcheck、golangci-lint 有什麼差別？你們 CI 跑哪個？

go vet 是官方基本檢查；staticcheck 是更深入的單一檢查器；golangci-lint 是聚合器，把數十個 linter（含前兩者）用一份設定檔統一執行與快取。CI 只跑 golangci-lint，規則寫在 repo 的 .golangci.yml，保證所有人與 CI 用同一套標準。

幾條你該知道的比對規則：

  • 帶 method 的 pattern（"GET /jobs"）只收該 method；path 存在但 method 不符時，ServeMux 自動回 405 Method Not Allowed。

1.3ResponseWriter 的規則與「header already written」

http.ResponseWriter 有一條看不見但鐵打的順序規則：status code 與 header 只能在第一次寫 body 之前送出。三個方法的關係：

  • w.Header().Set(...)：修改「即將送出」的 header，必須在 WriteHeader 之前。
  • w.WriteHeader(404)：送出 status line 與 header，只能呼叫一次。
  • w.Write(b)：寫 body；若還沒 WriteHeader，會隱式先送 200 OK。
陷阱

錯誤分支忘記 return 是 handler 最常見的 bug：錯誤回應送出後程式繼續往下跑，又寫了一次成功回應，client 收到兩段黏在一起的 body。Gin 的 c.Abort() 解決的就是同一類問題——標準庫裡它就是一個 return。

http.ListenAndServe(addr, mux) 只適合範例程式——它建出的 server 所有 timeout 都是零值（無限等待），一個惡意或故障的 client 慢慢送 header 就能永久佔住一條連線（slowloris 攻擊）。生產程式一律自己建 http.Server：

陷阱

WriteTimeout 到期時 server 直接切斷連線，你的 handler 不會收到任何通知，goroutine 還在傻傻計算。所以 server timeout 只是最後防線；per-request 的主動取消要靠第 2 週的 context。兩層都要有。

順帶理解 keep-alive：HTTP/1.1 預設同一條 TCP 連線可連續處理多個請求（省去反覆握手）。這就是為什麼需要 IdleTimeout——沒有它，每個「用完沒關」的 client 都在 server 上留一條殭屍連線。

Q1handler 裡先 w.Write(...) 再 w.WriteHeader(500) 會發生什麼事？為什麼？

顯示答案

client 仍收到 200：第一次 Write 已隱式送出 200 OK 與 header，之後的 WriteHeader 無效，server log 出現 "superfluous response.WriteHeader call"。HTTP 的 status line 在回應最前面，送出後物理上不可能收回。

Q2為什麼生產環境不能用 http.ListenAndServe(addr, mux) 這個便利函式？

顯示答案

它建立的 Server 所有 timeout 都是零值＝無限。慢速或惡意 client（slowloris）可以用極慢的 header/body 永久佔住連線，堆積到 fd 耗盡。必須自建 http.Server 設定 ReadHeaderTimeout / ReadTimeout / WriteTimeout / IdleTimeout。

Q3srv.Shutdown(ctx) 的 ctx 為什麼不能直接用 signal.NotifyContext 給的那個？

顯示答案

收到訊號的當下那個 ctx 已經處於 canceled 狀態，Shutdown 會立刻因 context 結束而返回，等於沒有 drain。必須用 context.Background() 派生一個帶新期限的 context 作為「最多等多久」的預算。

Q4liveness 和 readiness probe 各自回答什麼問題？shutdown 時誰先變化？

顯示答案

liveness＝「process 是否還活著、需不需要重啟」；readiness＝「現在能不能接流量」。shutdown 開始時應先把 readiness 翻成 503（流量被移走），liveness 維持 200（不需要重啟，只是在下班），然後才 drain 連線。

面試問答 · Week 1

Gin 的 Engine 和 http.Handler 是什麼關係？

*gin.Engine 實作了 ServeHTTP(w, r)，所以它「是一個」http.Handler，可以直接交給 http.Server 使用。Engine 在 ServeHTTP 裡做的事：從 sync.Pool 取出 gin.Context、依 radix tree 路由找 handler 鏈、依序執行 middleware 與 handler。也因此 Gin 專案照樣需要自己設定 http.Server 的 timeout 與 graceful shutdown——框架沒有替你處理 server lifecycle。

解釋 http.Server 的四種 timeout，以及只設 WriteTimeout 為什麼不夠。

ReadHeaderTimeout 限制收 header、ReadTimeout 限制收整個請求、WriteTimeout 限制讀完請求到寫完回應、IdleTimeout 限制 keep-alive 閒置。WriteTimeout 到期只是切斷連線，handler goroutine 不會被通知也不會停止，計算資源照樣被消耗；所以還需要 context 層級的取消（per-request deadline）讓 handler 主動中止工作。

SIGTERM 到 SIGKILL 之間，你的服務應該做哪些事？

順序：1) 把 readiness 翻為未就緒，讓 LB / kube-proxy 停止導入新流量；2) 關閉 listener 不收新連線（srv.Shutdown 做的事）；3) 等待進行中請求完成，上限設在 K8s terminationGracePeriodSeconds 之內（例如 30 秒的寬限期就給 20–25 秒）；4) 依序關閉背景 worker、flush log/metrics、關 DB 連線；5) 正常 exit 0。超時仍未完成的請求會被硬切，應記 log 供事後分析。

2.3goroutine 與 channel：規則先於直覺

go f() 啟動一個 goroutine，成本約幾 KB stack，開幾十萬個都行——問題從來不是「開太多」，而是「不知道它什麼時候結束、誰負責讓它結束」。寫下任何 go 關鍵字前先回答：這個 goroutine 在什麼條件下退出？誰等它？

channel 是 goroutine 之間的管道，行為規則值得背下來：

2.7Mutex、RWMutex 與 data race

data race 的定義：兩個 goroutine 同時存取同一記憶體位置，至少一個是寫，且沒有同步機制。race 是「未定義行為」——不是「偶爾算錯」，而是任何事都可能發生（撕裂讀取、map 直接 panic）。Go 的立場很清楚：有 race 的程式是錯的程式，沒有「良性 race」。

  • sync.Mutex：一次一人。保護「讀改寫」的完整過程，不是只保護寫。
  • sync.RWMutex:多讀單寫。讀遠多於寫時才值得，寫鎖會等所有讀鎖釋放。第 1 週的 memstore 已經用上。
  • sync.WaitGroup：等一組 goroutine 結束。Add 必須在 go 之前呼叫（在 goroutine 裡 Add 自己，Wait 可能先跑過）。
  • sync.Once：恰好執行一次，適合 lazy 初始化。
重點

race detector 只能抓到「實際發生過」的 race——測試沒跑到的交錯它看不到，所以它是必要條件不是充分條件。這也是為什麼 Makefile 的 test target 永遠帶 -race：讓每一次測試都是一次抽查。

Q1父 context 取消時子 context 會怎樣？反過來呢？

顯示答案

取消只向下傳播：父取消 → 整棵子樹取消（所有 Done channel 被 close）。子取消完全不影響父——這讓「為單一下游呼叫加 timeout」不會誤傷整個請求。

Q2對已 close 的 channel 分別做「送」和「收」會發生什麼？由誰 close 才對？

顯示答案

送 → panic；收 → 立刻得到零值且 ok=false（可無限收）。因此只能由發送方 close：發送方才知道「不再有資料」，接收方 close 等於在發送方腳下拆地板。

Q3什麼是 backpressure？job-api 裡它體現在哪兩個地方？

顯示答案

當下游消化不了時，向上游明確說「慢一點/現在不行」，而不是無限緩衝。體現：(1) queue 是有容量的 buffered channel；(2) Submit 用 select+default，滿了立刻回 ErrQueueFull → HTTP 429，把壓力誠實地推回 client。

Q4為什麼 wg.Add(1) 必須在 go func(){...}() 之前？

顯示答案

若在 goroutine 內部才 Add，主流程的 Wait() 可能在 Add 執行前就檢查計數器（此時為 0）直接通過，造成「還沒等就結束」的競態。Add 在 go 之前執行才能保證 Wait 看得到正確計數。

Q5本週 Runner 的 Job ctx 從 context.Background() 派生而不是從 request ctx，理由是什麼？什麼情況下你會選相反的設計？

顯示答案

Job 是「已被接受的非同步工作」，生命週期比 HTTP 請求長；client 斷線不應該讓已排隊的訓練工作消失。相反設計（掛在 request ctx 下）適合「結果只對這個請求有意義」的場景，例如即時查詢的 fan-out——沒人等答案就不必算下去。能講出這個 trade-off 就掌握了 context 的精髓。

面試問答 · Week 2

解釋 cancellation propagation：一個請求進來後取消是怎麼一路傳下去的？

server 為每個請求建立 ctx（掛在 server 的 base ctx 下）；handler 把 r.Context() 傳給 service，service 傳給 repository 與下游 client。client 斷線或 shutdown 時 server cancel 該 ctx，Done channel 關閉，所有以它為祖先的 goroutine 在下一個 select / 阻塞點收到通知並返回 ctx.Err()。關鍵設計：ctx 作為第一個參數逐層顯式傳遞，Go 不靠 thread-local。

你怎麼在生產環境發現並定位 goroutine leak？

發現：metrics 監控 go_goroutines（Prometheus Go collector 內建）持續攀升不回落。定位：對 /debug/pprof/goroutine?debug=1 抓兩個時間點的快照做差分，看哪個 stack 的數量在漲，該 stack 直接指出卡住的程式行。預防：測試掛 goleak、code review 盯每個 go 關鍵字的退出條件。

Mutex 和 channel 怎麼選？

經驗法則：保護「狀態」用 Mutex（如 map 的讀寫，臨界區小而清楚）；傳遞「所有權/工作流」用 channel（生產者消費者、pipeline、訊號廣播）。channel 不是比較高級——拿 channel 模擬鎖會又慢又難讀。Go 官方的說法：share memory by communicating 描述的是所有權轉移的場景，不是要你消滅所有鎖。

%w 保留錯誤鏈，errors.Is/As 仍能認出底層錯誤；%v 只留文字、斷開鏈。要讓呼叫端能據以分支處理時用 %w；底層錯誤是實作細節、不想變成公開契約時用 %v 斷開。

Q2errors.Is 和 errors.As 分別回答什麼問題？

顯示答案

Is：「鏈上有沒有這一個錯誤值？」——用於 sentinel。As：「鏈上有沒有這一型錯誤？有的話給我」——用於 typed error，取出後可讀其欄位。兩者都會沿 Unwrap 鏈走訪，取代只看表層的 == 與型別斷言。

Q3為什麼 500 回應的 body 不能放 err.Error()？那要怎麼查問題？

顯示答案

內部錯誤字串常含 SQL 片段、檔案路徑、內部服務位址——對攻擊者是偵察情報。正確做法：對外回固定的 "internal error" + requestId；完整錯誤鏈記進 log。查案流程是拿 client 提供的 requestId 去 log 系統撈出該請求全部記錄。

Q4「fail-fast 的 config」具體指什麼行為？為什麼在 K8s 上這反而是優點？

顯示答案

啟動時完整驗證所有設定，任何缺漏/非法立刻 exit 非零，絕不帶病服務。在 K8s 上，起不來的 Pod 呈現 CrashLoopBackOff、log 明確指出缺哪個變數，而且舊版本繼續服務（rolling update 不會把壞設定推滿全場）；帶病啟動反而讓錯誤設定悄悄接管流量。

面試問答 · Week 3

介紹你專案裡的錯誤處理策略。

三層：domain 定義 sentinel（ErrJobNotFound 等）與 typed error（ValidationError）；中間層用 %w wrap 加脈絡，形成可追溯的錯誤鏈；transport 層唯一一張映射表把 domain 錯誤翻成 HTTP status + 統一 error schema（code/message/requestId），未知錯誤一律 500 且不洩漏內部細節。強調兩個決策：映射集中一處（不散在各 handler）、repository 實作細節錯誤在 service 層斷鏈避免變成 API 契約。

結構化日誌比傳統日誌好在哪？你們 log 什麼、不 log 什麼？

結構化＝每筆 log 是鍵值對，查詢系統可以按欄位過濾聚合（status=500 且 duration_ms>1000），傳統字串只能 grep。我們每個請求記 request_id、method、path、status、duration；錯誤記完整錯誤鏈。不 log：token、密碼、Authorization header、request body 內容——log 系統的存取權限通常比資料庫寬，log 進去等於外洩。

retryable 與 non-retryable 錯誤怎麼區分？誰決定重試？

retryable＝失敗原因是暫時狀態（佇列滿、鎖競爭、網路抖動 → 429/503，且帶 Retry-After 更好）；non-retryable＝重試不會改變結果（資源不存在、驗證失敗 → 404/422）。決定權在「知道語義的一層」：server 用 status code 表達可重試性，client 據此實作退避重試。分類做得差，client 對 404 狂重試就是自製 DDoS。

Q1fake 和 mock 的差別？為什麼 Go 社群偏好 fake？

顯示答案

fake 是有簡化真實邏輯的實作（map 版 store），測試驗證「結果」；mock 預先寫死期望的呼叫序列，驗證「互動過程」。mock 與實作細節耦合，重構（不改行為）就大片翻紅；fake 讓測試只關心行為，更耐重構。mock 保留給「互動本身就是規格」的場合（例：必須恰好發送一次通知）。

Q2allocs/op 為什麼常比 ns/op 更值得關注？

顯示答案

堆配置的成本不只在配置當下——每個活著的物件都增加 GC 的掃描負擔，高配置率讓 GC 更頻繁地暫停/佔用 CPU，代價分攤到整個程式而不是這一行。ns/op 只看見局部延遲，allocs/op 預示全局的 GC 壓力。降低配置（重用 buffer、strings.Builder、預配 slice 容量）常是 Go 服務最有效的優化。

Q3heap profile 的 inuse_space 和 alloc_space 各回答什麼問題？

顯示答案

inuse_space：「現在誰佔著記憶體」——查記憶體洩漏、OOM。alloc_space：「從啟動以來誰累計配置最多」——查 GC 壓力來源（配置完馬上丟掉的東西在 inuse 裡看不到，在 alloc 裡現形）。查「越跑越肥」看前者，查「CPU 都被 GC 吃了」看後者。

Q4舉出兩種 flaky test 的成因與對應修法。

顯示答案

(1) 用 sleep 等非同步結果：機器慢一點就 fail → 改成 deadline 內輪詢，或注入假 clock。(2) 依賴 map 迭代順序：Go 刻意隨機化 → 斷言前排序。(3) 依賴 goroutine 調度順序 → 用 channel/WaitGroup 顯式同步。共同原則：測試的每一次執行必須走完全相同的邏輯路徑。

Q5為什麼 pprof 端點不能開在對外的 port 上？

顯示答案

heap dump 含有程式記憶體裡的實際資料（token、個資都可能在裡面），goroutine dump 暴露完整程式結構；CPU profile 端點還可被反覆呼叫造成負載。所以 debug server 只綁 localhost 或內部網路，需要時用 kubectl port-forward 連進去。

同一個模式的不同名字：包住一個可呼叫物，於前後插入橫切邏輯。HTTP 世界叫 middleware（func(Handler) Handler），RPC 世界叫 interceptor（第 7 週 ConnectRPC 的 UnaryInterceptorFunc），OO 世界叫 decorator。掌握一個，其餘只是簽名不同——這也是為什麼堅持用標準簽名寫 middleware：技能與程式碼都可以帶著走。

Protocol Buffers 與 API Contract

目標：學會 contract-first——先用 .proto 定義 API 契約，程式碼由契約生成，相容性由工具把關。本週定義的 JobService 契約，第 7 週會長成真正的 RPC 服務，之後 cluster-agent（W26）也走同一套。

6.1為什麼要 contract-first

你目前的 REST API 契約存在哪裡？答案是「散落在 handler 程式碼和大家的腦子裡」——欄位名打錯要到 runtime 才知道，改個型別沒有任何工具攔你。contract-first 把契約提升為一份獨立、可版本控制、可機器檢查的檔案（.proto），然後：

  • server 與 client 的型別、序列化、傳輸程式碼全部生成，兩端永遠一致。
  • 相容性檢查自動化：buf breaking 在 CI 擋下會弄壞既有 client 的改動。
  • Protobuf 的二進位編碼比 JSON 小且快，但這只是紅利——真正的價值是契約與演進紀律。

在本平台裡的分工（也是業界常見拓撲）：對外/瀏覽器 → REST 或 Connect；服務對服務 → RPC。Job API 的 REST endpoint 保留當相容層，內部通訊從第 7 週起走 ConnectRPC。

6.3Field number 與 wire compatibility：一輩子的承諾

理解相容性要先懂一點 wire format：序列化時欄位名稱不會出現在資料裡，只有 field number。name = 2 編碼成「tag 2 + 長度 + bytes」。這帶來兩個推論：

  • 改欄位名是安全的（wire 上根本沒有名字）——但會改生成程式碼，是原始碼層面的 breaking。
  • 改 field number 或型別是災難——舊 client 用舊號碼編碼，新 server 用新號碼解讀，資料會默默錯位或丟失。

相容性規則表，這是本週驗收的口試題：

操作安全？原因
新增欄位（新號碼）✅舊端不認識的欄位會當 unknown field 保留或忽略
刪除欄位 + reserved✅號碼被保留，永不再用
刪除欄位後重用號碼💥舊資料/舊 client 仍用該號碼送舊語義，新端會把它解讀成完全不同的欄位——資料靜默損毀

backward compatibility＝新 server 讀得懂舊 client 的資料；forward compatibility＝舊 client 讀得懂新 server 的資料（未知欄位忽略/保留）。protobuf 兩者兼顧的前提就是：號碼永不重用、只加不改。

重點

optional 在 proto3 的意義是「可以區分『沒設』與『零值』」（生成 Go 指標欄位 + Has 方法）。repeated 是列表、map<k,v> 是字典。什麼時候需要 optional？當「0 / 空字串」本身是合法業務值時——例如 quota 為 0 和沒設 quota 必須不同。

6.4API 設計慣例：resource name、pagination、filter

這些慣例大量借鑑 Google AIP（API Improvement Proposals），Kubernetes API 也是同一脈絡——現在學，第 11 週看 K8s API 會有既視感：

  • Resource name：資源用階層式名稱識別，如 tenants/team-a/jobs/123——名稱即位址，天然表達從屬關係。
  • Pagination：request 帶 page_size 與不透明的 page_token，response 回 next_page_token（空字串＝沒了）。token 是 server 的加密書籤，client 不該解讀——這讓 server 能自由改分頁實作。不要用 offset/limit 當公開契約：資料在翻頁間變動時 offset 會跳漏，且深分頁在 DB 上是 O(offset)。
  • Filter / order_by：字串欄位如 filter: "status = RUNNING"、order_by: "created_at desc"。一開始只支援你需要的最小子集，文件寫清楚。

Q1為什麼 protobuf 改欄位名安全、改 field number 是災難？

顯示答案

顯示答案

7.5Resilience：retry、backoff、jitter、circuit breaker

網路呼叫的失敗不是異常而是常態，韌性策略是一組互相配合的機制——單獨使用任何一個都可能幫倒忙：

Retry：只重試值得重試的

  • 只重試 unavailable、resource_exhausted 這類暫時性錯誤；invalid_argument、not_found 重試一萬次也一樣。
  • 只重試冪等的操作——GetJob 隨便重試；CreateJob 必須有 idempotency key（第 5 週）才能重試。
  • 設重試上限與總時間預算（受 deadline 約束），永遠不要無限重試。

Exponential backoff + jitter

為什麼一定要 jitter？想像下游閃斷 1 秒，1000 個 client 全部失敗、全部等「精確的 200ms」後同時重試——這叫 thundering herd，第二波齊射把剛復活的服務再打死。隨機化讓重試在時間軸上攤開。

Circuit breaker：停止對死者 CPR

當下游持續失敗，繼續重試只是浪費資源並拖慢自己（每個請求都等滿 timeout）。斷路器是一個三態狀態機：closed（正常放行，統計失敗率）→ 失敗率超閾值 → open（直接快速失敗，不再打下游）→ 冷卻期後 → half-open（放少量探測請求，成功則 closed、失敗則回 open）。價值：給下游喘息空間 + 讓自己快速失敗而不是被拖死。

Request hedging：知道就好，慎用

hedging＝不等失敗，p99 太慢就先發第二發請求，誰快用誰。它降尾延遲，但把系統負載直接乘倍，且要求嚴格冪等——負載高峰時 hedging 會把「慢」放大成「掛」。沒有成熟的負載保護前不要碰。

顯示答案

client 的 ctx deadline 會隨請求以 header（如 grpc-timeout）傳給 server，server 的 ctx 自動帶上，往下游呼叫時繼續傳——整條鏈共享同一份時間預算，上游取消也沿鏈傳播。deadline 應由最上游呼叫者設定（只有它知道使用者等多久是可接受的），中間層只傳遞或收緊、不放寬。

Q3為什麼 exponential backoff 一定要配 jitter？

顯示答案

沒有 jitter 時，同時失敗的大量 client 會在完全相同的時刻同步重試（thundering herd），每一波齊射都可能把剛恢復的下游再打倒。jitter 把重試時刻隨機攤開，讓下游面對的是平滑流量而不是脈衝。

Q4circuit breaker 的三個狀態與轉換條件？它和 retry 的關係？

顯示答案

closed（放行並統計失敗率）→ 失敗率超閾值轉 open（快速失敗、不打下游）→ 冷卻期滿轉 half-open（放少量探測）→ 探測成功回 closed、失敗回 open。retry 處理「偶發」失敗，breaker 處理「持續」失敗——retry 沒有 breaker 節制，會在下游故障時放大流量、拖垮自己。

Q5哪些 RPC code 適合自動重試？重試 CreateJob 的前提是什麼？

顯示答案

適合：unavailable、resource_exhausted（暫時性、與請求內容無關）。不適合：invalid_argument、not_found、already_exists、permission_denied（結果確定）。deadline_exceeded 要看預算還剩多少。CreateJob 有副作用，重試前提是冪等保障（idempotency key 或 server 端去重），否則逾時重送可能建立重複資源。

面試問答 · Week 7

你的服務怎麼同時提供 REST 和 RPC？兩邊怎麼避免邏輯重複？

分層的回報：REST handler 與 RPC handler 都只做「協定 ↔ DTO 轉換 + 錯誤映射」，然後呼叫同一個 application service。domain error 有兩張映射表（HTTP status / RPC code），業務邏輯零重複。connect-go 的 handler 本身是 http.Handler，兩者掛在同一個 chi router、共用同一條 middleware 鏈與同一個 http.Server 的 lifecycle。

設計一個對外的重試策略，你會考慮哪些維度？

五個維度：(1) 可重試性分類——只重試暫時性錯誤碼；(2) 冪等性——非冪等操作要求 idempotency key；(3) 節奏——exponential backoff + full jitter，設單次與總預算上限，受整體 deadline 約束；(4) 全局保護——circuit breaker 防止對持續故障的下游堆積重試，配合 server 端 rate limit；(5) 可觀測——重試次數進 metrics，讓「重試在掩蓋的故障」被看見。並強調 retry budget 概念：重試流量占比超過某閾值就該熔斷，因為那已經是故障放大器。

server streaming 連線的生命週期管理要注意什麼？

每條 stream 是一個長壽 goroutine + 一份訂閱資源，必須三件事齊備：(1) select 監聽 ctx.Done——client 斷線、deadline、server shutdown 都由此通知；(2) defer 清理訂閱，防 goroutine/記憶體 leak；(3) 資料源關閉（channel close）時正常結束 stream。另外 server graceful shutdown 要考慮 stream 可能永遠不結束——Shutdown 的 drain 對長連線要設上限並主動取消 base context。

8.2Non-root、read-only 與容器裡的訊號

容器安全的兩條底線設定，一次講清楚「為什麼」：

  • Non-root user：容器裡的 root ≈ 有限制的宿主 root。一旦發生 container escape 漏洞，容器內的 uid 就是你在宿主上的 uid——nonroot 讓逃逸的攻擊者只是個普通使用者。distroless 的 :nonroot tag 內建 uid 65532。程式面的配合：監聽 8080 而不是 80（非 root 不能綁 1024 以下的 port）。
  • Read-only root filesystem：程式運行時不該改自己的檔案系統——這讓「攻擊者植入後門/寫入惡意檔案」變得困難。K8s 端設 securityContext.readOnlyRootFilesystem: true；程式若需要暫存檔，明確掛一個 emptyDir 到 /tmp。你的 Job API 目前無檔案寫入，正好驗證這件事。

8.3Tag、digest、SBOM 與弱點掃描

tag 是可變的指標，digest 是內容的雜湊。myapp:v1.2.3 今天和明天可以指向不同內容（tag 可被覆蓋推送）；myapp@sha256:abc... 永遠精確指向一份 bytes。因此：

  • 人類溝通用 tag，生產部署用 digest（或至少 immutable tag 政策）——「昨天還好好的」的排查前提是你確切知道跑的是哪份 image。
  • latest tag 在生產是禁語：不可重現、無法回滾到「上一個 latest」。

Week 9 · Phase 3

Kubernetes Workload、Config 與 Resource Management

目標：把 Job API 部署進 kind，並建立 Kubernetes 的核心心智模型——「宣告期望狀態，controller 負責收斂」。本週還要刻意弄壞它三次：OOMKilled、CrashLoopBackOff、ImagePullBackOff 各來一遍，除錯能力是練出來的。

9.1Pod、ReplicaSet、Deployment：三層各管一件事

Kubernetes 的一切從這條鏈開始，三層各自的職責要能一句話說出：

  • Pod：最小部署單位＝一組共享網路/儲存的容器。Pod 是牲口不是寵物——它會死、會被驅逐，且死了不會自己復活。你幾乎永遠不直接建 Pod。
  • ReplicaSet：「維持 N 份這樣的 Pod」。Pod 掛了就補一個。它只管數量，不管版本更新。
  • Deployment：管理 ReplicaSet 的版本演進。改 image 時它建立新 RS、逐步放大、縮小舊 RS——rolling update 與 rollback 都是它的把戲。

其餘 workload 型別按「Pod 的個性」選擇：StatefulSet（Pod 有穩定身分：固定名字 pod-0/pod-1、各自的儲存——資料庫用）、Job（跑完就結束，管完成次數與重試）、CronJob（排程的 Job）、DaemonSet（每個 node 恰好一份——log agent、CNI 用）。你的 Job API 是無狀態服務 → Deployment。

重點

整個 K8s 只有一種工作原理：你把期望狀態寫進 API（宣告式），controller 迴圈不斷比較「期望 vs 實際」並收斂差距。Deployment/RS/Job 全是這個模式的實例。第 11 週起你會自己寫 controller，屆時回頭看這週會豁然開朗。

9.2完整 manifest 逐段讀：Deployment + Service + Config

本週的主檔案。每一段都有存在的理由，照著打一遍、再逐段對照說明：

三個第一次見面的角色：

  • Service：Pod 的 IP 會變，Service 給一個穩定的虛擬 IP + DNS 名（platform-api.default.svc），把流量分給 selector 選中的、readiness 通過的 Pod。第 1 週寫的 readyz 在這裡接上了流量控制。
  • ConfigMap / Secret：第 3 週「config 來自環境變數」的 K8s 版供給端。兩者機制相同，差在用途宣告與治理：Secret 可以被獨立 RBAC 控管、加密存放（etcd encryption at rest 需另外開啟——Secret 預設只是 base64，不是加密！）。
  • Probe 三兄弟：readiness＝能不能接流量（失敗→從 Service 摘除）；liveness＝需不需要重啟（失敗→殺掉重啟）；startup＝慢啟動程式的緩衝（成功前 liveness 不生效）。
陷阱

liveness probe 別檢查外部依賴（DB、下游服務）！DB 抖一下 → 全部 Pod liveness 失敗 → 全部同時重啟 → 雪崩。liveness 只回答「我這個 process 卡死了嗎」；依賴健康屬於 readiness（暫時摘流量，等恢復）。這是生產環境最常見的自殘設定。

9.3Requests、limits 與 QoS：OOMKilled 的原理

兩個數字管兩件不同的事：requests 是排程用的保證（scheduler 找一個「還有這麼多沒被認領」的 node），limits 是執行時的天花板。而 CPU 和 memory 的天花板行為完全不同：

超過 limit 時後果
CPU被 throttle（壓速）變慢，不會死。但 throttle 會推高尾延遲
Memory被 OOM killer 殺掉容器直接死，事件顯示 OOMKilled，然後依 restartPolicy 重啟

實務配置策略（也是本專案的預設）：memory 的 request = limit（記憶體不可壓縮，超賣的代價是隨機 OOM）；CPU 只設 request、不設 limit（CPU 可壓縮，空閒時讓 Pod 用好用滿，避免無謂 throttle）——這也是 Google 與多數 SRE 文獻的建議起點。requests/limits 的組合決定 QoS class：兩者相等＝Guaranteed（最後被驅逐），只設 requests＝Burstable，全沒設＝BestEffort（第一個被驅逐）。

Go 程式的特別注意：container 的 CPU limit 和 GOMAXPROCS 脫鉤——Go runtime 看到的是 node 的核心數。設了 CPU limit 卻不管 GOMAXPROCS，會造成嚴重 throttle（runtime 以為有 16 核，實際配額 1 核）。解法：Go 1.25+ runtime 已自動感知 cgroup 配額；舊版本用 automaxprocs 套件或手動設環境變數。

9.4Rolling update 與 rollback

改了 image 之後發生的事：Deployment 建新 RS，按 maxSurge（最多多開幾個）與 maxUnavailable（最多少幾個可用）的節奏，新 Pod ready 一個、舊 Pod 殺一個。零停機的成立條件是一整條鏈，缺一環就會有毛刺，全部是你前八週做過的事：

  1. 新 Pod 的 readiness 通過才接流量（probe 正確）。
  2. 舊 Pod 收 SIGTERM 先翻 readyz、drain 連線（W1 graceful shutdown）。
  3. SIGTERM 真的到得了程式（W8 exec form）。
  4. drain 時間 < terminationGracePeriodSeconds。

rollback 之所以快，是因為舊 ReplicaSet 沒被刪除（保留 revisionHistoryLimit 個），undo 只是反向再做一次 rolling。這再次解釋為什麼要 digest：「回滾到上一版」的前提是上一版指得回去。

9.5排程進階：affinity、taint、topology spread

scheduler 預設只看資源是否夠。四個機制讓你表達更多意圖，方向不同要分清：

  • nodeAffinity：Pod 挑 node——「我要去有 GPU 的節點」（W30 的主角）。
  • podAffinity / podAntiAffinity：Pod 挑鄰居——anti-affinity 最常用：「我的兩個副本不要同一台 node」，單機故障不全滅。
  • taint + toleration：node 挑 Pod（方向相反！）——node 宣告「閒人勿近」（taint），只有帶通行證（toleration）的 Pod 能上。GPU 節點、專用節點的標準做法。
  • topologySpreadConstraints：把副本平均攤在 zone / node 上，是 anti-affinity 的一般化，大規模時效能更好。

PriorityClass 補最後一塊：資源不夠時誰先被犧牲。平台元件（controller、gateway）設高 priority，批次工作設低——高優先 Pod 排不進去時，scheduler 會驅逐低優先的騰位子（preemption）。

Q1liveness 和 readiness 失敗的後果各是什麼？為什麼 liveness 不該檢查資料庫連線？

顯示答案

readiness 失敗→從 Service endpoints 摘除（不接流量，Pod 活著）；liveness 失敗→容器被殺掉重啟。DB 抖動時若 liveness 檢查 DB：所有 Pod 同時判死、同時重啟，服務全斷且重啟風暴加劇故障——依賴問題應由 readiness 暫時摘流量，等依賴恢復自動回歸。

Q2CPU 和 memory 超過 limit 的行為有何不同？由此推出什麼配置策略？

顯示答案

CPU 可壓縮：超限被 throttle，變慢不死。memory 不可壓縮：超限被 OOM kill。策略：memory request=limit（不超賣，避免隨機 OOM）；CPU 設 request 不設 limit（避免無謂 throttle 推高延遲）。Go 程式另需注意 GOMAXPROCS 與 cgroup 配額對齊。

Q3taint/toleration 和 nodeAffinity 的方向差異？GPU 節點通常怎麼配？

顯示答案

taint 是 node 排斥 Pod（沒 toleration 的進不來）；nodeAffinity 是 Pod 選擇 node。GPU 節點兩者並用：taint 擋住一般 Pod 佔用昂貴節點，GPU 工作負載帶 toleration + nodeAffinity/nodeSelector 主動選過去——「別人進不來，我一定要去」。

面試問答 · Week 9

解釋一次零停機 rolling update 需要哪些條件配合。

從流量視角走一遍：新 Pod 要有 readiness probe，通過才進 Service endpoints；舊 Pod 收到 SIGTERM 時先把 readyz 翻 503 讓流量摘除、再 drain 進行中請求（graceful shutdown），且 SIGTERM 到得了程式（exec form ENTRYPOINT、PID 1）；drain 上限小於 terminationGracePeriodSeconds；maxSurge/maxUnavailable 控制節奏保證容量。再加一句：驗證方式是 update 期間持續打流量觀察錯誤率，而不是「看起來沒事」。

Deployment 和 StatefulSet 怎麼選？

問三個問題：Pod 需要穩定的網路身分嗎（pod-0 的名字重啟後不變）？需要各自綁定的持久儲存嗎（pod-0 永遠拿 volume-0）？啟動/縮容需要順序保證嗎？任一個 yes → StatefulSet（資料庫、Kafka、有狀態叢集成員）。全部 no → Deployment。常見誤區：只是「有寫檔案」不代表需要 StatefulSet——共享儲存或外部資料庫的無狀態服務仍用 Deployment。

QoS class 有哪三種？驅逐順序怎麼決定？

Guaranteed（所有容器 requests=limits）、Burstable（設了部分）、BestEffort（全沒設）。node 資源吃緊觸發 kubelet eviction 時，BestEffort 先走，其次超用 requests 最多的 Burstable，Guaranteed 最受保護；PriorityClass 再疊加一層——scheduler 的 preemption 與 kubelet 的 eviction 都會參考 priority。平台核心元件應設 Guaranteed + 高 priority。

10.4RBAC：誰、能對什麼、做什麼

  • ServiceAccount：程式的身分（人類走 OIDC/憑證）。每個 Pod 掛一個 SA，token 自動注入——你的 controller（W11 起）就是用它跟 API server 說話。
  • Role：一組權限規則，限定單一 namespace；ClusterRole：全叢集（或用於 node、PV 這類無 namespace 的資源）。
  • RoleBinding：把 Role（或 ClusterRole）授予主體，效力限於該 namespace；ClusterRoleBinding：全叢集生效。

一個多租戶 namespace 的安全基線，你會配哪些東西？

Kubernetes API Object 與 Client

目標：把 K8s 從「kubectl 的對象」變成「你程式的資料庫」。理解 API object 的解剖學（GVK、metadata、spec/status）、樂觀並發，並用 client-go 寫出第一個操作叢集的程式。這章是之後所有 controller/operator 的地基。

11.1GVK：API 世界的座標系

K8s API 裡每種物件由三元組定位——Group / Version / Kind：apps/v1 Deployment、core group（寫作空字串）的 v1 Pod、以及你第 14 週要發明的 platform.example.com/v1alpha1 Tenant。對照 YAML 就是 apiVersion: apps/v1（group/version）+ kind: Deployment。

與 GVK 對應的是 GVR（Group/Version/Resource）：resource 是 HTTP 路徑上的複數小寫名（deployments）。Kind 是型別的名字，Resource 是端點的名字，兩者的對應表由 RESTMapper 透過 API discovery（kubectl api-resources 看到的東西）取得。多數時候框架幫你轉換，但讀錯誤訊息時分得清 GVK/GVR 會省很多時間。

Subresource：同一個物件的獨立操作面——pods/log、pods/exec、deployments/scale，以及最重要的 */status：spec 與 status 分開的端點、分開的 RBAC 授權，這是「使用者寫 spec、controller 寫 status」分權的機制基礎。

11.2物件解剖學：metadata、spec、status 與那三個神秘欄位

每個 K8s 物件都是同一個形狀：metadata（身分與索引）+ spec（使用者的意圖）+ status（controller 的觀察）。spec/status 的分工是整個系統的憲法：使用者永遠不寫 status，controller 永遠不改 spec。metadata 裡三個新手最困惑的欄位，直接背下語義：

欄位是什麼什麼時候變用途
uid這個「物件實例」的唯一 ID永不變；刪掉重建同名物件會拿到新的 uid區分「同名的前世今生」；ownerReference 用它防呆
generationspec 的版本號只有 spec 變更時 +1（status/metadata 變更不動）搭配 status.observedGeneration 判斷「controller 處理到哪一版意圖」（W16 主角）
resourceVersionetcd 的變更序號任何寫入都變樂觀並發（11.3）與 watch 的續傳書籤

Labels vs annotations：labels 是可查詢的索引（selector 用它選物件，值有字元限制）；annotations 是掛在物件上的便條（工具存狀態、說明文字，不能用來選）。經驗法則：會拿來「篩選一群物件」的放 label，其他一律 annotation。

11.3樂觀並發：conflict 是日常，不是錯誤

K8s API 沒有鎖。並發控制靠 resourceVersion 的樂觀並發：update 請求帶著你讀到的 resourceVersion，若 etcd 裡已經不是這個版本（別人先寫了），API server 回 409 Conflict。正確反應不是報錯，而是重讀、重改、重送：

Update 之外的第二條路是 Patch——只送你要改的部分，衝突面小得多。三種 patch 要能分辨：JSON patch（操作序列：add/replace/remove，精確但囉嗦）、merge patch（送一個片段蓋上去，list 是整個替換）、strategic merge patch（K8s 特製：懂 list 的合併語義，如 containers 按 name merge）。

第三條路是現代首選 Server-Side Apply（SSA）：你宣告「我管理的欄位長這樣」，API server 記錄每個欄位的管理者（managedFields），不同管理者互不覆蓋、同欄位衝突會明確報錯。這是 controller 時代的 apply 語義，W15 的 CreateOrUpdate 與它相輔相成。

重點

什麼時候用哪個：讀改寫整個物件 → Update + RetryOnConflict；只碰自己的一小塊（加 annotation、改 replicas）→ Patch；「我持續管理這個物件的這些欄位」（controller 的常態）→ SSA。

11.4List-Watch：K8s 的心跳

你不會想每秒 List 全部物件來知道「有沒有變」——API server 提供 watch：一條長連線，物件每次變更推一個事件（ADDED / MODIFIED / DELETED）。標準模式是 list-watch：先 List 取得全量 + 當下的 resourceVersion，再從這個版本開始 Watch 增量——不漏也不重。

但 watch 連線會斷（網路、API server 重啟、etcd 壓縮舊版本導致書籤失效 410 Gone），斷了要重新 list-watch。這套「斷線重連、全量增量對齊」的苦工，正是第 12 週 informer 要替你封裝的東西——先在這裡懂原理，下週才知道 informer 在解決什麼。

Q2409 Conflict 發生的機制是什麼？正確的處理模式？

顯示答案

顯示答案

顯示答案

多個管理者長期共管同一物件的欄位所有權問題。SSA 讓 API server 記錄每個欄位由誰管理（managedFields）：你 apply 時只聲明自己管的欄位，別人的欄位不受影響；兩個管理者搶同一欄位會得到明確的 conflict 而不是靜默互相覆蓋。這是 controller 與 kubectl apply、HPA 等多方共存的基礎。

面試問答 · Week 11

解釋 spec 與 status 分離的設計哲學，以及它如何體現在 API 機制上。

spec 是期望狀態（人寫），status 是觀察到的實際狀態（controller 寫），分離讓「意圖」與「現實」可以獨立演進、比較差距即是 reconcile 的工作定義。機制上：status 是獨立 subresource——寫 status 走 /status 端點、需要獨立的 RBAC 權限、不影響 generation（generation 只追蹤 spec 變更），因此使用者與 controller 各自的寫入互不干擾，也不會互相觸發對方的處理循環。

watch 機制怎麼保證 client 不漏事件？resourceVersion 太舊會怎樣？

list-watch 模式：List 回傳全量與一個 resourceVersion 快照點，Watch 從該版本開始收增量，銜接處無縫。斷線後從最後收到的版本續傳；但 etcd 只保留有限的歷史（會壓縮），版本太舊時 watch 收到 410 Gone——client 必須重新 List 對齊全量再 Watch。informer 把這整套（含 410 重列）封裝掉了，這也是生產程式碼永遠用 informer 而不是裸 watch 的原因。

Lister 是 cache 的唯讀查詢介面：lister.Deployments(ns).Get(name) 完全不打 API server——controller 的讀取幾乎全走這裡，這是 controller 能夠高頻 reconcile 而不壓垮 API server 的原因。代價是下一節的主題：cache 可能是舊的。

12.2為什麼事件不直接處理：workqueue 的三個理由

新手的直覺是「在 UpdateFunc 裡直接做事」。錯，而且錯得有教育意義。正確結構是 event handler 只把 key（namespace/name）丟進 workqueue，由固定數量的 worker 從 queue 取 key 處理。三個理由：

  • 解耦速度：事件可能爆量湧入（重列時全部物件都來一次 Add），handler 必須瞬間返回；真正的處理由 worker 按自己的節奏消化——這是第 2 週 backpressure 的重演。
  • 合併重複（coalescing）：workqueue 對「已在佇列中的相同 key」去重——一個物件連續改十次，worker 可能只需處理一次，因為處理時讀的是 cache 裡的最新狀態，不是事件裡的舊快照。
  • 失敗重試：rate-limiting queue 內建指數退避——處理失敗呼叫 AddRateLimited(key)，這個 key 會在遞增的延遲後重新出隊；成功則 Forget(key) 清除失敗計數。
重點

由此推出 controller 的第一原則：queue 裡放 key 不放物件、事件只是「該看看這個 key 了」的門鈴。門鈴響幾次不重要、順序也不重要，開門時看到的現場（cache/實際狀態）才是事實。這就是 level-triggered（第 13 週正式命名）。

12.4Resync、亂序與 stale cache：三個必須內化的性質

  • Resync：每隔 resync period，informer 把 cache 裡全部物件重新走一遍 UpdateFunc（old==new）。它不是重新 List API server，而是本地重放——用途是兜底：萬一哪次事件處理失敗又沒重試到，resync 保證終究會再看到每個物件。這也意味著你的 reconcile 必然會被無變化地重複呼叫——冪等不是美德是義務。
  • 事件會亂序、會合併、至少一次：watch 重連時可能把 MODIFIED 合併；同 key 去重讓你「錯過」中間態。controller 不准依賴事件順序與次數，只准依賴「當下讀到的狀態」。寫進你的技術筆記標題：事件不等於狀態。
  • Cache 是舊的（stale read）：你剛 Update 完一個物件，立刻從 lister 讀——很可能讀到舊版（watch 事件還沒回來）。controller 的邏輯必須容忍「讀到過去」，做法是把決策設計成收斂式：讀到舊的 → 這輪做了重複/無效操作 → 冪等吸收 → 下個事件再收斂。

Q1為什麼 event handler 只 enqueue key，而不是把物件或「做什麼」放進 queue？

顯示答案

三個原因：(1) key 可去重合併，物件快照不能（會處理過時資料）；(2) worker 處理時從 cache 讀最新狀態，天然跳過中間態；(3) 「做什麼」是由「當下狀態 vs 期望狀態」推導的，不是由事件決定的——事件只是觸發器。這就是 level-triggered 設計。

Q2WaitForCacheSync 在等什麼？不等會發生什麼？

顯示答案

等 informer 完成啟動時的首次全量 List 並填滿本地 cache。不等就開工：lister 讀到半份世界，controller 可能誤判「物件不存在」而做出破壞性決策（例如 GC controller 誤刪「孤兒」）。任何依賴 cache 的邏輯都必須在 sync 完成後才啟動。

Q3queue 的 Done、Forget、AddRateLimited 各是什麼語義？

顯示答案

Done(key)：本次處理結束（無論成敗），釋放「處理中」標記——處理期間再次 Add 的同 key 會在 Done 後重新出隊，保證不並發處理同一 key。Forget(key)：清除 rate limiter 的失敗計數（成功時呼叫）。AddRateLimited(key)：按指數退避延遲後重新入隊（失敗時呼叫）。三者組合構成 at-least-once + 退避重試。

Q4你剛 Update 一個物件，馬上從 lister 讀卻是舊值——為什麼？怎麼設計才安全？

顯示答案

lister 讀的是本地 cache，更新要等 watch 事件回來才會反映——寫後讀有天然延遲。安全設計不是「等 cache 新鮮」，而是讓邏輯收斂冪等：基於舊值做的操作要嘛無害重複（create 已存在→吸收 AlreadyExists）、要嘛下輪事件會修正。需要強一致讀的罕見場景直接 Get API server（付出負載代價）。

Week 13 · Phase 4

Controller Reliability

目標：把上週「會動」的 controller 練成「壞不了」的 controller。本週全是分散式系統的核心觀念在 controller 上的落地：level-triggered、冪等、最終一致、leader election——這些正是 Operator 面試的主戰場。

13.1Level-triggered：K8s 世界觀的正式命名

兩種反應模式的對比，值得用開關和燈想像：

  • Edge-triggered（邊緣觸發）：對「變化的瞬間」反應——「開關被按下時開燈」。錯過那個瞬間（當機、斷線、事件合併），狀態就永遠錯了。
  • Level-triggered（電平觸發）：對「當下的狀態」反應——「只要開關在 ON、燈就該亮」，每次檢查都重新對齊。錯過任何事件都沒關係，下一次檢查會把世界修正。

K8s controller 一律 level-triggered：事件（門鈴）觸發 reconcile，但 reconcile 讀的是完整的當下狀態並向期望狀態收斂。這解釋了上週所有「怪癖」為什麼是 feature：事件可以丟、可以重複、可以亂序、resync 可以無腦重放——因為正確性不依賴事件，依賴收斂。這也是「eventual consistency」在 controller 語境的含義：任何時刻的狀態可能落後，但停止擾動後系統收斂到期望狀態。

重點

檢驗你的 reconcile 是否真的 level-triggered 的思考實驗：「controller 停機一小時後重啟，期間世界隨便變——重啟後不做任何特殊補償，跑幾輪 reconcile 能否完全恢復正確？」能，才合格。

13.2冪等的具體技法

「reconcile 必然被重複呼叫」上週已確立，本節給出讓重複無害的標準手法：

  • Create 吸收 AlreadyExists：if apierrors.IsAlreadyExists(err) { /* 視為成功，繼續 */ }——重複的建立操作是常態不是錯誤。
  • 寫之前先 diff：讀當下物件、算出期望的樣子，一樣就不寫。省 API 負載，更重要的是不觸發無意義的 watch 事件（否則你自己的更新喚醒自己的 reconcile，形成熱迴圈）。
  • Conflict 用 RetryOnConflict 或改用 SSA/Patch（W11 的三條路在這裡復用）。
  • 副作用要有去重鍵：發通知、呼叫外部 API 這類不可自然冪等的動作，用「已處理標記」（status 欄位或 annotation）防止重複執行。

13.3Leader election：多副本但一人做事

controller 要高可用就得跑多副本，但兩個副本同時 reconcile 同一物件會互相打架（重複建立、交錯更新）。解法不是分工而是選主：多副本競爭一把鎖，只有 leader 工作，其他人待命，leader 掛了幾秒內有人接手。

機制：競爭者輪流嘗試更新一個 Lease 物件（coordination.k8s.io/v1），寫入自己的身分與續約時間——本質是「用 K8s API 的樂觀並發實作分散式租約鎖」。三個時間參數的關係要懂：leaseDuration（鎖多久沒續約算失效）> renewDeadline（leader 最多花多久續約）> retryPeriod（競爭者多久試一次）。

陷阱

失去領導權時直接退出 process 是業界慣例（K8s 自己的元件也這樣）。看似粗暴，實則最安全：你無法保證手上進行到一半的工作能瞬間全部剎車，而「舊 leader 還在動、新 leader 也開工」的 split-brain 比重啟貴得多。K8s 會把你拉起來當一個乾淨的競爭者。

Week 14 · Phase 5

Kubebuilder 與 CRD API Design

目標：用 Kubebuilder 開出你自己的 Kubernetes API——Tenant。本週重點不是「跑起來」而是「設計得像 K8s 原生資源」：schema 驗證、printer columns、status subresource，讓 kubectl 使用者感覺不出這是第三方資源。

14.1CRD 是什麼：把你的概念放進 API server

CustomResourceDefinition 讓你擴充 K8s API：註冊一個 CRD 之後，API server 就會提供 /apis/platform.example.com/v1alpha1/tenants 端點——CRUD、watch、RBAC、kubectl 全部自動獲得，etcd 幫你存資料。你不用寫任何伺服器程式碼就有了一個宣告式 API。

而 Operator = CRD（自訂 API）+ Controller（讓 API 成真的程式）。分工和 Deployment 完全同構：CRD 定義 spec/status 的形狀，controller 負責把 spec 的意圖變成現實、把現實寫回 status。你前三週親手搭的 informer/workqueue 就是 controller 的引擎室——Kubebuilder 把它們裝進整流罩。

Kubebuilder 的三個機制先認識：marker（// +kubebuilder:... 註解，是「用註解寫設定」——controller-gen 讀取它們生成 CRD YAML 與 RBAC）；deepcopy（K8s 物件在 cache 與各處傳遞需要深拷貝，zz_generated.deepcopy.go 自動生成——這就是為什麼每次改 types 要跑 make generate）；make manifests（重新生成 CRD YAML）。

14.2Tenant API：types.go 逐段設計

把計畫裡的 YAML 反推成 Go 型別，每個 marker 都是一條 API 契約：

marker 逐個拆解它「買到了什麼」：

  • validation 系列：變成 CRD 的 OpenAPI schema，API server 在寫入前就拒絕非法資料——controller 永遠不會看到 pods=-1 的 Tenant。能在 schema 層擋的就不要進 webhook，能在 webhook 擋的就不要進 controller。
  • subresource:status：開啟 /status 端點——沒有它，spec 與 status 就是同一個寫入面，W11 講的分權機制蕩然無存。忘記這個 marker 是新手第一大坑（symptom：status 更新默默消失或覆蓋 spec 變更）。

Structural schema 是 CRD 的入場券：每個欄位都要有明確型別、不能有模糊的任意結構（除非明確標記 preserveUnknownFields）。這是 API server 能做 validation、pruning（剪掉未知欄位）與 SSA 欄位管理的前提。Kubebuilder 生成的 schema 天然 structural——但你要知道這個詞，面試會問。

Q1CRD 和 Operator 的關係？只裝 CRD 不跑 controller 會怎樣？

顯示答案

CRD 只是「在 API server 註冊一種新資料型別」——apply CRD 後你就能 CRUD Tenant 物件，資料乖乖存進 etcd，但什麼事都不會發生：spec 只是願望清單。Operator = CRD + controller，controller 才是把願望變現實的勞動力。這個「API 與實現分離」正是宣告式系統的本質。

Q2validation 放 CRD schema、webhook、controller 三個位置的取捨？

顯示答案

優先序由早到晚：schema（宣告式、無運行成本、拒絕發生在寫入前、kubectl explain 可見）→ webhook（需要跨欄位/查外部狀態的邏輯驗證，仍在寫入前，但引入可用性依賴）→ controller（最後防線，只能事後把錯誤反映在 status，資料已經進 etcd）。原則：能早擋就早擋——越晚發現，使用者回饋迴路越長。

15.2OwnerReference：資源樹的血緣系統

ownerReference 是 metadata 裡的一個列表：「我屬於誰」。它買到三件事：

  • 級聯刪除（garbage collection）：owner 被刪，K8s 的 GC controller 自動刪掉所有 children——Tenant 刪除時七件套自動消失，不用寫任何清理程式碼（外部資源除外，W17）。
  • Owns watch 的路由：上一節講的「子變動找父」。
  • uid 綁定：reference 記錄 owner 的 uid——同名新 Tenant 不會誤認舊 children（W11 的 uid 伏筆回收）。
陷阱

cluster-scoped 資源（如 Namespace）的 owner 只能是 cluster-scoped 資源——這正是我們把 Tenant 設計成 cluster-scoped 的原因之一。反過來，namespaced 資源不能 own 其他 namespace 裡的資源。掛錯 scope 的 ownerReference，GC 會直接把 child 刪掉（它認為 reference 無效）——「我的資源一建立就被神秘刪除」的經典謎題。

Q1Owns() 做了什麼？「手動刪除子資源會自動補回」的完整機制鏈是？

顯示答案

Owns 註冊對該型別的 watch，事件到達時沿 ownerReference 解析出 owner、把 owner 的 key 入隊。補回鏈：刪 Namespace → DELETE 事件 → 找到 owner Tenant → Tenant reconcile → ensureNamespace 發現不存在 → CreateOrUpdate 重建。注意 controller 從頭到尾不「知道」是刪除——它只是重新對齊期望狀態（level-triggered）。

Q2CreateOrUpdate 的 mutate 函式有什麼書寫紀律？

顯示答案

(1) 「宣告期望」而非「打補丁」——每次完整重申你管理的欄位值，不能依賴「第一次已經設過」；(2) 不碰不屬於你的欄位（別人或系統管理的，如 status、clusterIP）；(3) immutable 欄位只在建立分支設定；(4) ownerReference 在 mutate 裡設（保證重建時也有）。違反 (1) 會導致漂移不收斂，違反 (2) 會和其他管理者打架。

Q3Tenant 刪除時 controller 的 Reconcile 只做了 IgnoreNotFound，children 是誰清的？

顯示答案

K8s 內建的 garbage collector controller：它 watch 所有資源，發現物件的 ownerReference 指向已不存在的 uid 時將其刪除（預設 background 級聯）。所以「叢集內、有 ownerReference 的」清理是免費的；需要自己動手的是外部資源與跨叢集資源——那是 W17 finalizer 的領域。

Q4為什麼 RBAC 用 marker 生成而不是手寫 YAML？

顯示答案

權限需求由程式碼決定（你 Get/Create 什麼就需要什麼權限），marker 讓兩者物理上相鄰、同 PR 演進、make manifests 機械同步——手寫 YAML 的漂移模式是「程式加了新資源操作、Role 忘了加」（上線才 Forbidden）或「權限加了從不收」（越積越肥違反最小權限）。marker 是「權限即程式碼」的實踐。

面試問答 · Week 15

描述你的 Tenant Operator 的 reconcile 流程與冪等保證。

Reconcile 讀取 Tenant，對 spec.namespaces 逐一執行七個 ensure（Namespace、Quota、LimitRange、SA、Role、RoleBinding、NetworkPolicy），每個 ensure 用 CreateOrUpdate：Get 現況 → mutate 重申期望 → 有 diff 才提交。冪等來自三處：CreateOrUpdate 的 diff 判斷（重複執行 op=unchanged）、mutate 的宣告式寫法（結果只依賴 spec 不依賴歷史）、ownerReference 的 uid 綁定（不會誤操作別人的資源）。任何失敗返回 error 由框架退避重試，成功路徑不留狀態機。

ownerReference 和 finalizer 的分工是什麼？

兩者都參與刪除，方向相反：ownerReference 是「我死時把我的 children 帶走」（GC 自動級聯，適用叢集內資源）；finalizer 是「我死前必須先完成某些事」（阻擋刪除直到清理完成，適用外部資源、需要順序的清理）。Tenant Operator 兩個都用：七件套靠 ownerReference 免費清理，外部 registry 記錄靠 finalizer 保證先清（W17）。常見誤區是用 finalizer 去清 ownerReference 已經會清的東西——多餘且增加 stuck terminating 風險。

16.2ObservedGeneration：「你處理到我哪一版了？」

回收 W11 的伏筆。使用者剛改了 spec（generation 從 3 → 4），看到 status 顯示 Ready=True——這個 True 是舊 spec 的 True 還是新 spec 的？沒有 observedGeneration 就無法回答。協議很簡單：

Phase 是給人掃一眼的摘要，conditions 是給人與機器細讀的明細。鐵律：phase 由 conditions 推導，永遠不要獨立維護兩套判斷（會漂移）：

status 寫入的兩個工程細節：

  • Conflict 處理：status update 一樣會 409（main container 與你並發改同一物件）。標準解法：status 更新失敗直接返回 err 讓框架重跑整輪 reconcile——比在原地 retry 乾淨，因為重跑會重讀最新物件。

Q1kubectl delete 之後、物件真正消失之前，發生了什麼？

顯示答案

若 finalizers 非空：API server 只設定 deletionTimestamp，物件進入 Terminating 但仍完整存在且可讀。各 finalizer 的主人 controller 看到 deletionTimestamp 後執行各自清理、完成後移除自己的 finalizer。最後一個 finalizer 移除時，API server 才從 etcd 刪除。刪除是「協商」出來的，不是命令。

Q2為什麼 finalizer 必須在建立外部資源「之前」掛上？

顯示答案

考慮 crash 窗口：先建外部資源再掛 finalizer，中間 crash → 外部資源已存在但沒有 finalizer 保護 → 使用者刪除物件直接成功 → 外部資源永久孤兒。先掛 finalizer 再建，最壞是「有 finalizer 沒資源」——冪等的 cleanup 對不存在的資源直接成功，無害。順序把「孤兒窗口」變成「空跑窗口」。

Q3清理外部資源時對方回 404，正確處理是什麼？為什麼？

顯示答案

視為成功（目標狀態「不存在」已達成）。cleanup 會因重試、controller 重啟而重複執行——第一次成功刪除後，第二次必然 404；把 404 當錯誤會讓 finalizer 永遠移不掉，物件永久 Terminating。冪等刪除的通用定義：以「結果狀態」而非「操作是否執行」判定成功。

Q4手動 patch 掉 finalizer 的真實代價是什麼？什麼情況下可以接受？

顯示答案

代價：該 finalizer 承諾的清理永遠不會發生——外部資源孤兒化（持續計費、佔用名額、安全殘留）。可接受的情況：確認清理已不可能或已人工完成（外部系統已下線、資源已手動刪除），且留下記錄。它是「接受損失」的決定，不是除錯技巧。

18.1Admission webhook：寫入路徑上的攔截點

一個寫入請求進 API server 的旅程：認證 → 授權 → mutating webhooks（可修改物件）→ schema validation → validating webhooks（只能說 yes/no）→ 存入 etcd。webhook 就是 API server 在這兩個位置回呼你的 HTTPS 服務。什麼時候需要它們（而不是 schema）：

  • Mutating / defaulting：預設值需要「邏輯」——例如 defaultLimits 沒填時，按 quota 的 1/4 計算；或自動補 label。
  • Validating：跨欄位規則（defaultLimits.cpu 不得超過 quota.cpu）、需要查外部狀態的規則（namespace 名稱不得與其他 Tenant 重疊）。
陷阱

webhook 是寫入路徑上的同步依賴：failurePolicy: Fail 時 webhook 服務掛掉＝所有 Tenant 寫入被擋（安全但可用性差）；Ignore 則反過來（可用但漏檢）。生產抉擇：validating 常用 Fail，但 webhook 服務本身要高可用；絕不要對核心資源（pods、所有 namespaces）掛 Fail 的 webhook 而不做排除——webhook 掛掉會讓整個叢集癱瘓（包括你要修它的那次部署！慣例：webhook 排除自己所在的 namespace）。cert 管理交給 cert-manager，本機開發用 envtest 免 cert。

18.2版本演進：v1alpha1 → v1beta1 的實際操作

假設 Tenant 要改一個欄位設計（例：quota 從 flat 結構變成 per-namespace 列表）。破壞性變更的正確流程不是「改掉」，而是兩版並存 + 自動轉換：

  1. 新開 v1beta1 型別（新設計），舊 v1alpha1 保留。
  2. CRD 標記一個 storage version（etcd 裡實際存的版本，通常選新版）；其餘版本 served（可讀寫）。
  3. Conversion webhook 負責雙向翻譯：任何 client 用任何版本讀寫，API server 都經 webhook 轉換——舊 client 完全無感。

Kubebuilder 的 hub & spoke 模式讓 N 個版本只需要 N-1 對轉換（而不是 N×N）：選一個版本當 hub（通常是 storage version），其他版本只實作「與 hub 互轉」：

面試級的理解：conversion 的難點不是程式碼而是語義——新版本表達力更強時，「新 → 舊」必然有損。策略：降級規則文件化，或把裝不下的資訊塞進 annotation 做 round-trip 保真。這也反過來教你 API 設計：欄位語義想清楚再進 beta，conversion 的複雜度是設計倉促的利息。

顯示答案

Week 19 · Phase 6

Platform API Server 與 TenantService

目標：把「使用者 → 平台」的入口建起來。TenantService 是一個 ConnectRPC 服務，背後不是資料庫，而是 Kubernetes API——建立 Tenant＝建立 CR，然後由 Operator 收斂。本週的核心是這個「API 疊在 API 上」的邊界設計。

19.1架構定位：為什麼不讓使用者直接 kubectl

你的 Operator 已經能用 kubectl 操作，為什麼還要一層 Platform API？四個理由，也是所有內部平台（IDP）的共同答案：

  • 認證模型不同：終端使用者用公司 SSO（OIDC），不該人手一份 kubeconfig。
  • 授權粒度不同：「team-a 的 admin 只能動 team-a」這種業務授權，K8s RBAC 表達起來很彆扭（W20 的雙層模型）。
  • 介面穩定性：CR 的 schema 是平台內部實作，Platform API 的 proto 才是對使用者的契約——兩者可以獨立演進（DTO ≠ CR）。
  • 聚合與加值：審計、配額計算、跨 cluster 視圖（W25 起），這些都不是單一 CR 能回答的。
重點

注意「寫入即返回」：CreateTenant 在 CR 寫入成功後就回應，不等 Operator 完成——因為完成可能要幾十秒，且宣告式系統裡「已受理」與「已生效」本來就是兩件事。這就是 long-running operation 的處理模式：回傳資源本體（含 status），讓 client 以 Watch 或輪詢追蹤進度，「Ready 了沒」由 W16 的 conditions/observedGeneration 協議回答——你前幾週的設計在這裡全部接上。

19.2DTO ↔ CR 映射：兩個世界的翻譯規則

proto 的 Tenant message 和 CRD 的 Tenant 型別長得像，但絕不共用——它們的變更節奏、驗證規則、欄位語義都不同。翻譯層的紀律：

三條翻譯紀律：內部機制不外洩（resourceVersion、finalizer、managedFields 都不出現在 proto）；status 只讀（DTO→CR 方向絕不碰 status）；錯誤翻譯——K8s API 的錯誤也要走翻譯（AlreadyExists → already_exists、Invalid → invalid_argument 帶欄位路徑、Forbidden → permission_denied），第 3/7 週的錯誤映射表家族再添一張。

Field mask 基礎：更新操作的「只改這幾個欄位」語義。完整實作複雜，本週採簡化版：UpdateTenant 的 request 帶 update_mask（如 "display_name,quota"），server 只把 mask 內的欄位從 DTO 覆蓋到讀出的 CR 上再寫回——避免「讀改寫」把使用者沒動的欄位也蓋掉。

19.3Watch、pagination 與 audit metadata

WatchTenant（server streaming，W7 的技能）背後直接用 controller-runtime client 的 Watch 或 informer：把 CR 的變更事件翻譯成 proto event 推給 client。注意翻譯層的「事件不等於狀態」（W13）——推送的是物件當下快照，不保證每個中間態都送達。

Pagination：K8s List 支援 limit + continue token——正好和 W6 的 page_token 設計對齊，把 continue 當作不透明 token 傳給 client 即可（記得 base64 包一層，宣示其不透明性）。

Audit metadata：每個變更操作記錄「誰、何時、做了什麼、從哪來」。最小實作：從 auth interceptor 的 Identity 取 who，寫進兩個地方——結構化 log（audit=true 標記，W21 接到 audit log 管線）與 CR 的 annotation（platform.example.com/last-modified-by），後者讓 kubectl 使用者也看得到。

任務本週實作清單

  • tenant.proto 定義（沿 W6 慣例：分頁、UNSPECIFIED enum、Timestamp），buf lint/breaking 過。
  • TenantService 五個 RPC 實作；platform-api 的 ServiceAccount 只授予 tenants CRUD（不給 cluster-admin！）。
  • CreateTenant 支援 Idempotency（W5 的 key 模式或天然冪等：同 id 已存在且 spec 相同 → 回 200 現況）。
  • integration test：Create → Watch 直到 READY → Delete → Watch 到消失（跑在 envtest 或 kind 上，operator 同時運行）。

✅ 第 19 週驗收清單

  • curl 打 CreateTenant → kubectl 看得到 CR → operator 收斂 → GetTenant 回 READY。
  • DTO 不含任何 K8s 內部欄位；K8s 錯誤被正確翻譯成 RPC code。
  • 重送同一個 CreateTenant 不會出錯也不會重建。
  • audit：每個變更在 log 與 annotation 都留下 who/when。

「API on top of Kubernetes」這種架構的優缺點？

優點：拿到宣告式核心的全部紅利——期望狀態存 etcd、收斂由 operator 負責、watch 免費、審計與 RBAC 內建，平台 API 層薄到只剩認證授權與翻譯。缺點與對策：K8s API 不是高吞吐 OLTP 資料庫（etcd 寫入有限）——適合控制面（低頻、高價值的資源操作），不適合資料面（每秒千筆的業務交易，那要真資料庫）；查詢能力弱（label selector 而已）——複雜查詢要另建索引或 cache。結論：控制面用 CR，資料面別硬塞。

UpdateTenant 怎麼避免「讀改寫」互相覆蓋？你有哪幾層防護？

三層：(1) field mask——client 聲明只改哪些欄位，server 只覆蓋 mask 內欄位，縮小衝突面；(2) K8s 樂觀並發——寫回 CR 時 resourceVersion 不符會 409，翻譯成 aborted 讓 client 重試（或 server 端 RetryOnConflict 重讀重套 mask）；(3) 對特定高衝突欄位改用 patch 語義。加一句區分：field mask 解決「別蓋到我沒改的」，樂觀並發解決「兩人同時改同一個」——正交的兩個問題。

Week 20 · Phase 6

OIDC、JWT 與 Authorization

目標：把 W5 的「靜態 token」換成真的企業級認證：OIDC 登入、JWT 驗證、JWKS 輪替，並建立 tenant-aware 的雙層授權模型。這章的每個概念都是資安面試的常客。

20.1OAuth 2.0 vs OIDC：授權框架 vs 身分層

一句話分工：OAuth 2.0 回答「這個 app 能存取什麼資源」（authorization），OIDC 在其上加一層回答「使用者是誰」（authentication）。OIDC = OAuth 2.0 + ID token（JWT 格式的身分證明）+ 標準化的 discovery/userinfo 端點。

Authorization Code flow（web 應用的標準流程，配 PKCE）：瀏覽器被導到 IdP（Dex/Keycloak）→ 使用者登入 → IdP 帶著一次性 code 導回 → client 後端用 code + PKCE verifier 換 token。設計精髓：token 永遠不經過瀏覽器的 URL（code 才經過，且一次性、短命、綁 PKCE），密碼永遠只給 IdP。

20.2JWT 驗證：五個必查項，一個都不能少

JWT = header.payload.signature（base64url），簽章保證未被竄改，但內容人人可讀（不要放秘密！）。API server 收到 token 的驗證清單：

五個必查項與「不查會怎樣」：signature（不查＝任何人可偽造任意身分）；issuer（不查＝別家 IdP 簽的 token 也收）；audience（不查＝發給別的服務的 token 拿來打你——confused deputy 的一種）；expiry（不查＝洩漏的 token 永久有效）；演算法（庫要鎖定預期演算法，歷史上的 alg=none 攻擊）。JWKS 輪替：IdP 換簽名鑰匙時 JWKS 端點同時提供新舊公鑰（kid 區分），庫按 token header 的 kid 選鑰——所以永遠不要把公鑰寫死在 config。

Refresh token 基礎：access token 短命（分鐘級，洩漏損害有限），refresh token 長命且只對 IdP 使用（換新 access token）。API server 完全不碰 refresh token——那是 client 與 IdP 之間的事。

20.3雙層授權：應用層規則 + K8s RBAC 兜底

計畫裡那張圖是本週的靈魂，展開講：

兩個必懂的授權概念：

  • Confused deputy：有權力的中間人（platform-api 有寫 CR 的權力）被無權力的請求者騙去做事。防禦就是「以請求者的身分做決策」——authorize 用的是 token 裡的 claims，而不是「platform-api 自己有沒有權限」。凡是「服務代替使用者操作」的地方都要問一次這個問題。
  • RBAC vs ABAC：role-based（身分 → 角色 → 權限，簡單可審計）vs attribute-based（依屬性規則計算：資源 owner、時間、來源 IP……，表達力強但難推理）。我們的模型：RBAC 為骨（roles claim），tenant 歸屬是一個屬性判斷——實務系統多是這種混合體。

Q1ID token 和 access token 各給誰用？拿 ID token 打 API 錯在哪？

顯示答案

ID token 是給 client 的「登入證明」（audience=client ID）；access token 是給資源伺服器的「存取憑證」（audience=API）。ID token 打 API 時 audience 驗證應該失敗——API 若收了，等於接受「發給別人的憑證」，任何拿到使用者 ID token 的第三方 app 都能冒用其身分打你的 API。

Q2JWT 驗證的五個必查項？漏查 audience 的具體攻擊情境？

顯示答案

signature（JWKS 公鑰）、issuer、audience、expiry、演算法白名單。漏查 audience：攻擊者是「服務 B」的合法使用者，拿著發給 B 的 token（同一個 IdP 簽的，簽章/issuer/expiry 全部合法）打你的服務 A——A 只驗簽章就會放行。audience 是「這張票是進你家的」的唯一證明。

Q3JWKS 與 key rotation 的機制？為什麼不能把公鑰放 config？

顯示答案

IdP 在 JWKS 端點發布當前有效的公鑰集（各有 kid），token header 標記簽發用的 kid，驗證方按 kid 取鑰並快取、未知 kid 時重抓。輪替時新舊鑰並存，舊 token 自然過期後移除舊鑰——零停機。公鑰寫死 config 的後果：IdP 輪替（例行或緊急撤換）那一刻全部驗證失敗，或更糟——為了不炸而永不輪替。

Q4解釋 confused deputy，以及你的平台哪裡有這個風險、怎麼防。

顯示答案

高權限中間人被低權限請求者利用去做請求者自己無權做的事。我們的 platform-api 就是 deputy：它的 SA 有寫所有 Tenant CR 的權力，若授權判斷寫錯（或只檢查「有登入」），tenant-a 的使用者就能透過它動 tenant-b。防禦：每個操作以「請求者的 claims」做 tenant-aware 授權，且 deputy 自身權限最小化（只有 Tenant CRUD），雙層縱深。

從瀏覽器點登入到 API 回應，把完整的認證授權流程講一遍。

Week 21 · Phase 6

Metrics、Tracing、Logging 與 Audit

目標：把平台從「能動」升級到「看得見」。三支柱各司其職：metrics 回答「系統整體如何」、trace 回答「這個請求去哪了」、log 回答「當時發生什麼」——最後用 SLO 把「多好才算好」寫成數字。

21.1Metrics：RED、USE 與 Prometheus 四型別

兩個口訣對應兩類對象：RED 給服務（Rate 請求率、Errors 錯誤率、Duration 延遲——使用者感受的三要素）；USE 給資源（Utilization 使用率、Saturation 飽和度/排隊、Errors——CPU、記憶體、佇列這類「有容量的東西」）。你的 API 用 RED，worker queue 用 USE（queue depth 就是 saturation）。

Prometheus 四種 metric 型別與選型：

Cardinality（基數）是 metrics 的頭號殺手：每一組不同的 label 值組合都是一條獨立時間序列。label 放 user_id、tenant_id（無界值域）→ 序列數爆炸 → Prometheus OOM。規則：label 只放有限集合（procedure、code、node），無界的識別資訊屬於 log 和 trace，不屬於 metrics。設計每個 metric 時先心算「這會產生幾條序列」。

21.2OpenTelemetry tracing：跨服務的因果鏈

一個 CreateTenant 慢，慢在哪？log 只有點，trace 給你整條因果鏈：trace 由 span 組成（span=一段有始有終的工作，帶 parent 關係），跨服務靠 context propagation——W3C traceparent header 隨請求傳遞，下游把自己的 span 掛到同一條 trace 上。

計畫裡的目標鏈路：Platform API span → K8s API request span → Operator reconcile 的 log 關聯。最後一段是關鍵技巧：operator 的 reconcile 不在同一個請求鏈上（非同步），無法直接掛 span——解法是log correlation：platform-api 把 trace_id 寫進 CR 的 annotation，operator 的 reconcile log 帶上它；查問題時用 trace_id 同時撈出 trace 與兩邊的 log。三支柱在 trace_id/request_id 上會師（W3 的 request ID 中介層是這一切的前身）。

21.3Audit log 與 SLO：面向責任與面向目標

Audit log 與一般 log 的差異在「不可否認性需求」：誰在何時對什麼做了什麼——安全事件的證據鏈。實作紀律：獨立的 logger 標記（audit=true）、欄位固定（actor、action、resource、tenant、result、request_id）、只記事實不記 payload（W3 的敏感資料紀律）、保存期限比一般 log 長。K8s API server 自己也有 audit log——平台層與叢集層兩份對照，就是完整的責任鏈。

SLI → SLO → error budget，用你的 API 實際定義一次：

  • SLI（指標）：可用性 = 非 5xx 請求占比；延遲 = p99 < 500ms 的請求占比。皆可由 RED metrics 算出。
  • SLO（目標）：30 天窗口，可用性 99.5%、延遲 SLI 99%。
  • Error budget（預算）：100% − SLO = 0.5%，30 天約 3.6 小時的「可以壞」額度。預算花完 → 停功能上線、還可靠性債——這是 SLO 的全部意義：把「要多穩」變成工程與產品都認帳的合約，而不是「越穩越好」的空話。

三支柱（metrics/traces/logs）各自的長處與典型的查案流程？

metrics：聚合視角，便宜、可告警，回答「有沒有事、多嚴重」；traces：單請求的跨服務因果鏈，回答「慢/錯在哪一段」；logs：任意細節的事件記錄，回答「當時到底發生什麼」。查案流程即三段跳：告警（metrics 的 SLO burn rate）→ 從 dashboard 的異常時段抓 exemplar trace 定位到服務與 span → 用 trace_id/request_id 撈該請求的 log 看細節。設計時反向要求：log 帶 correlation ID、trace 有取樣但異常全收、metrics 桶界配 SLO。

SLO 和「監控告警」的關係？error budget 政策實際上怎麼運作？

SLO 把告警從「症狀導向」（CPU 高了）升級為「目標導向」（再這樣燒 2 小時就違約）。告警建立在 burn rate 上：短窗口快燒（1 小時燒掉 2% 預算）→ page；長窗口慢燒 → ticket。預算政策是組織合約：預算充足時放手上功能，預算耗盡時凍結 release、資源轉向可靠性工作——它把「穩定 vs 速度」的爭論變成事先同意的自動決策。沒有 budget 政策的 SLO 只是牆上的數字。

Week 22 · Phase 7

Gateway API

目標：把第 10 週的「最小 HTTPRoute」升級成完整的多租戶入口設計：TLS、跨 namespace 授權（ReferenceGrant）、route 狀態判讀。核心視角：Gateway API 是「標準介面」，實作是可替換的 controller——這個分離本身就是它最重要的設計。

22.1資源模型：三層角色的權力劃分

複習並加深第 10 週的三層，這次從「誰擁有哪個資源」的角度：

資源擁有者回答的問題
GatewayClass基礎設施供應商/平台用哪個實作（Envoy Gateway、Istio、雲 LB）

Route attachment 是三層的接合機制，且是雙向同意：Route 用 parentRefs 聲明想掛哪個 Gateway；Gateway 用 listeners[].allowedRoutes 聲明接受誰（同 namespace？帶特定 label 的 namespace？全部？）。單方面想掛是掛不上的——這是多租戶安全的關鍵：租戶不能擅自佔用平台的入口。

22.3TLS、GRPCRoute 與「標準 vs 實作」

TLS 三種模式：Terminate（gateway 解密，內部走明文或 mesh mTLS——最常見）；Passthrough（gateway 不解密直通後端，用於後端要自己驗 client cert）；以及 gateway 到 backend 的重加密（BackendTLSPolicy，較新）。我們的平台：edge Terminate + 內部交給下週的 Istio mTLS——各管一段。

GRPCRoute：gRPC 的專用 route（按 service/method 比對）。你的 ConnectRPC 服務走 HTTP，用 HTTPRoute 以 /job.v1.JobService PathPrefix 比對即可——但要知道 GRPCRoute 存在及其理由（gRPC 的 method 級路由、header 語義）。

標準與實作的區別（面試高頻）：Gateway API 只是 CRD 規格 + 一致性測試；實際搬流量的是各家 controller（Envoy Gateway、Istio、Cilium、雲供應商）。同一份 manifest 理論上可攜——但各家的 GatewayClass 參數、擴充欄位（policy attachment）不可攜。選型時分清「你依賴的是標準欄位還是某家的擴充」。

Week 23 · Phase 7

Istio Traffic、mTLS 與 AuthorizationPolicy

目標：理解 service mesh 在平台裡買到的三件事——服務身分與 mTLS（不改一行程式碼的加密與認證）、L7 授權、流量治理（canary、retry、熔斷）。原則：懂機制與代價，而不是背 YAML。

23.1Mesh 的解剖：control plane、data plane 與服務身分

Data plane：每個 Pod 旁的 Envoy proxy（sidecar 模式）攔截進出流量——加密、路由、重試、遙測都發生在這裡，應用程式無感。Control plane（istiod）：把你的設定編譯成 Envoy 配置下發、簽發 workload 憑證。Ambient mode 是新架構：不注入 sidecar，L4 由節點層的 ztunnel 處理、L7 需要時才上 waypoint proxy——資源開銷更低，概念先知道即可，本週實作用 sidecar。

mesh 的地基是服務身分（service identity）：每個 workload 依其 ServiceAccount 獲得 SPIFFE 格式的身分證書（spiffe://cluster.local/ns/team-a-prod/sa/model-service），由 istiod 自動簽發輪替。這個身分是 mTLS 與授權的主詞——「誰在呼叫」從此有密碼學保證，而不是看 IP。

重點

mesh 與你前面所學的關係：W20 的 OIDC 解決「使用者是誰」（人對服務），mesh 的 mTLS 解決「服務是誰」（服務對服務）；W10 的 NetworkPolicy 是 L3/4 的「能不能連」，AuthorizationPolicy 是 L7 的「能做什麼操作」。四者是互補的層，不是替代品。

23.2mTLS 與 AuthorizationPolicy：零信任的两件套

遷移策略是 PERMISSIVE 的存在理由：先全網 PERMISSIVE（mesh 內自動 mTLS、mesh 外明文也通）→ 用遙測確認沒有殘餘明文流量 → 切 STRICT。直接上 STRICT 會把還沒進 mesh 的呼叫者全部切斷。

注意語義細節：一旦某 workload 有了 ALLOW policy，沒被任何規則匹配的請求默認拒絕（從「全開」翻轉成「白名單」）。principals 依賴 mTLS——所以 AuthorizationPolicy 要配 STRICT 模式才是完整的（PERMISSIVE 下明文流量沒有身分可查）。

23.3流量治理：canary、retry、熔斷——以及 retry storm 演習

Istio 的流量規則由 VirtualService（路由與拆分——與 Gateway API 的 HTTPRoute 職能重疊，整合趨勢是 mesh 也實作 Gateway API，兩者你都要看得懂）與 DestinationRule（目的地策略：subset 定義、連線池、熔斷）組成：

Retry storm 演習（本週的重頭戲）：現在你的呼叫鏈上有三層重試的可能——應用（W7 的 client retry）、mesh（VirtualService retries）、還有使用者的手動重刷。讓 model service 開始回 500，觀察放大效應：應用重試 3 次 × mesh 重試 2 次 = 一個使用者請求變 6 個打到後端。調整原則：

  • 重試只放一層（建議：簡單的網路級重試給 mesh，語義級重試給應用——但兩者不同時開大）。
  • mesh retry 只對「安全的」錯誤（connect-failure、503+冪等方法），並設 perTryTimeout。
  • 熔斷（outlierDetection）是 storm 的斷路器——踢掉壞 endpoint 讓重試有意義而不是重複撞牆。

Q1mesh 的服務身分從哪來？為什麼說它比 IP 可信？

顯示答案

istiod 依 workload 的 ServiceAccount 簽發短命 X.509 憑證（SPIFFE ID 編碼 ns+sa），sidecar 間 mTLS 握手時互驗。IP 在 K8s 裡是短暫且可被偽造來源的（NAT、IP 重用），憑證身分有密碼學保證且與部署身分（SA）綁定——授權規則因此能寫「哪個服務」而不是「哪個網段」。

Q2PERMISSIVE 模式存在的意義？直接全網 STRICT 會怎樣？

顯示答案

PERMISSIVE 同時接受 mTLS 與明文，是遷移的橋：mesh 內的呼叫自動升級成 mTLS，還沒注入 sidecar 的舊 client 不會被切斷。直接 STRICT：所有 mesh 外/未注入的呼叫者（含還沒重啟注入的 Pod、外部探針）瞬間全部連線失敗。正確流程：PERMISSIVE → 遙測確認明文歸零 → STRICT。

Q3AuthorizationPolicy 和 NetworkPolicy 的分工？

顯示答案

NetworkPolicy：L3/4，由 CNI 執行，「哪些 Pod 之間能建立連線」——不懂 HTTP，也不需要 mesh。AuthorizationPolicy：L7，由 Envoy 執行，基於密碼學身分（principal）與請求內容（method/path/header）決定「這個操作允不允許」。縱深：NetworkPolicy 縮小連得上的面，AuthorizationPolicy 管連上之後能做什麼——前者失效（CNI 不支援）後者仍在，反之亦然。

Q4retry storm 是怎麼形成的？你的三條對策？

顯示答案

多層重試相乘：應用×mesh×客戶端手動，一個失敗請求以指數倍數重擊已經不健康的後端，把局部故障放大成全面雪崩。對策：(1) 重試收斂到單層並限定可重試錯誤類型（冪等/連線級）；(2) perTryTimeout + 總預算，防重試吃光 deadline；(3) outlierDetection/circuit breaker 快速把壞 endpoint 移出池，配合 W7 的 backoff+jitter 攤平時間軸。並用 mesh 遙測監控 retry 比率作為預警。

面試問答 · Week 23

平台要不要上 service mesh？你的決策框架？

先列 mesh 買到什麼：全網 mTLS 與身分（合規常是硬需求）、L7 授權、無侵入的流量治理與遙測。再列代價：每 Pod 的 sidecar 資源與延遲、控制面運維複雜度、除錯多一層（是我的 bug 還是 Envoy 設定？）、升級手術。決策：服務數少、單租戶 → 用 library（W7 的 client 治理）+ NetworkPolicy 就夠；多租戶、多團隊、合規要求 service-to-service 加密 → mesh 的統一治理划算，且 ambient mode 正在壓低 sidecar 稅。關鍵句：mesh 是組織規模的解，不是技術炫技的解。

Gateway API 和 Istio VirtualService 的關係與未來？

功能重疊（路由、拆分、header 操作），出身不同：VirtualService 是 Istio 私有 API，Gateway API 是 K8s 官方標準。趨勢是合流：Istio 已實作 Gateway API（gateway 與 mesh 兩種 profile），新專案建議南北向直接用 Gateway API，東西向（mesh 內)按需選——標準欄位優先、Istio 專屬能力（複雜 fault injection、mirror 等）再用其 API 補。可攜性策略：依賴標準欄位的部分未來可換實作，用了私有欄位就是選了廠商——知情地選。

24.2CSI 與 Rook：儲存的 control plane

CSI（Container Storage Interface）是 K8s 與儲存系統之間的標準插件介面——provision/attach/mount 三段生命週期都由 CSI driver 實作。你在 W10 用的 local-path、雲上的 EBS/PD、以及 Ceph，接進 K8s 的方式都是 CSI driver + StorageClass。兩個進階能力本週練：volume expansion（StorageClass 設 allowVolumeExpansion: true，改 PVC 的 requests 就地擴容）與 snapshot（VolumeSnapshot 資源 → 從快照還原新 PVC——備份與實驗分支的基礎）。

Rook 是「用 Operator 跑 Ceph」——你 Phase 5 學的那套模式的工業級範例：CephCluster CR 宣告期望（幾個 mon、OSD 放哪些盤），Rook operator 負責部署、擴縮、修復。讀 Rook 的 CRD 設計（CephCluster/CephBlockPool/CephObjectStore 的 spec/status/conditions）本身就是 operator 設計的進修課。

24.3模型 artifact 的儲存設計

AI 平台的核心決策之一：模型檔案與 runtime image 分離。image 裡只有推論引擎（vLLM、Triton、你的 Go 服務），模型權重放 object storage，Pod 啟動時（或 init container）拉取。為什麼：

  • 尺寸與節奏：模型動輒數 GB～數百 GB，塞進 image 讓 build/push/pull 全部爆炸；且模型更新頻率 ≠ 引擎更新頻率，綁在一起互相拖累。
  • 版本語義：模型要獨立的版本線（model-x@v3 配 runtime@1.2 或 1.3）；分離後這是個配置組合，綁定則是笛卡兒積的 image 矩陣。
  • W28 的伏筆：KServe 的 storageInitializer 正是這個模式的標準化——storageUri: s3://models/x/v3。

Data locality 的一般化理解：資料與運算的距離決定冷啟動與延遲——這也是 W30 GPU 排程要把「模型已快取的節點」納入考量的原因。储存設計從來不只是「放哪裡」，而是「用的時候要移動多少 bytes」。

25.1Cluster 也是一種資源：Registry 的設計

幾個欄位背後的必學概念：

  • Cluster identity：名字不夠——kind cluster 重建後名字一樣但已是「另一個叢集」。用 kube-system namespace 的 UID 當叢集指紋（它與叢集共生死），註冊時記錄、之後每次通訊校驗，防「換了叢集還當舊的管」。

Q1為什麼叢集斷聯要標 Unknown 而不是 Failed？

顯示答案

「聯繫不上」與「它壞了」是兩件事：網路分區時叢集本身可能健康地跑著工作負載。標 Failed 可能觸發錯誤的自動化（把工作負載遷走 → 恢復後雙份運行）。Unknown 誠實表達資訊狀態，後續動作（等待、告警、遷移）成為明確的政策決定而非誤判——這是分散式系統對 partition 的基本尊重（CAP 的日常版）。

Q2cluster identity 為什麼不能只用名字？用什麼？

顯示答案

名字是註冊方取的標籤，叢集销毀重建後名字可以沿用，但它已是全新的叢集（空的、不同憑證、不同狀態）——把新叢集當舊的會做出災難性假設（以為 workload 還在）。用叢集內生的不變指紋：慣例是 kube-system namespace 的 UID。註冊時記錄、通訊時校驗，不符即拒絕並要求重新註冊。

Q3對 member 叢集的 rest.Config 為什麼必須設 Timeout 與 QPS？

顯示答案

遠端叢集會慢、會半死（TCP 通但 API 卡）。無 timeout 的呼叫讓 reconcile worker 無限期掛在一個壞叢集上——幾個壞叢集就吃光 worker pool，健康叢集也沒人管了（noisy neighbor 拖垮控制面）。timeout 保證單次呼叫有上界，QPS/Burst 防止對單一叢集的重試風暴，兩者一起實現「故障隔離：一個叢集的問題不擴散」。

面試問答 · Week 25

設計 cluster registry 時，spec 和 status 各放什麼？判斷準則？

準則沿用 K8s 憲法：spec=人的意圖與註冊事實（連線模式、租戶歸屬、憑證引用、placement 標籤），status=系統探測的觀察（版本、容量、心跳時間、phase/conditions）——全部可以從現實重新探測重建。反例警示：把「探測到的版本」放 spec（會漂移）、把「期望的租戶歸屬」放 status（丟了就沒了）。另外憑證永遠是 Secret 引用不內嵌，讓 RBAC 能對憑證與註冊資訊分離授權。

Week 26 · Phase 8

Cluster Agent 與 ConnectRPC Streaming

目標：實作 agent 模式——member 叢集裡跑一個 cluster-agent，主動向管理面建立長連線：回報心跳與容量、接收部署指令、回報結果。W7 的 streaming 技能在這裡從練習變成架構核心。

26.1Reverse connection：為什麼方向反過來就解決了一切

Agent 模式的精髓一句話：連線由 member 主動向外（outbound）建立，指令沿著這條既有連線反向流回。買到的東西：member 在 NAT/防火牆後也能被管理（只需出站 443）；管理面不需要持有任何 member 的 kubeconfig（agent 在叢集內用本地 ServiceAccount 操作）；憑證風險從「集中一大包」變成「每個 agent 只有自己的身分」。這是 GitOps pull 模式、Rancher、OCM 的共同拓撲。

26.2可靠訊息的四件套：seq、ack、重連、離線佇列

長連線一定會斷，訊息一定會重複——所以協定要自帶四個機制（全部是 W7/W13 概念的應用）：

  • Sequence number：每條指令帶遞增序號。agent 記住「已處理到第 N 號」，重連時帶上 N，管理面從 N+1 續傳——斷線不丟指令。
  • Ack（ReportResult）：管理面把指令標記為「已確認」的唯一依據是收到結果回報，不是「已送出」。逾時未 ack → 重發（at-least-once）。
  • 重複訊息處理：at-least-once 必然重複，agent 端以 instruction id 去重——已處理過的直接重放上次結果。更根本的保險：apply 操作本身冪等（SSA，W27），重放無害。
  • Offline queue：agent 斷線期間，管理面把待發指令排進 per-cluster 佇列（有上限！超過就標 cluster 需要全量重同步而不是無限堆積）；agent 側反向同理——結果回報在斷線時本地暫存，重連後補送。
陷阱

lastSeq 必須持久化（agent Pod 會重啟）且「執行→回報→記錄 seq」的順序決定語義：先記 seq 後執行=最多一次（crash 丟指令）；先執行後記 seq=至少一次（crash 重做指令）。選 at-least-once + 冪等執行——這是和 W12「Done 之前 crash 會重放」完全相同的抉擇。能把這兩件事連起來講，說明你真的懂了。

26.3Certificate bootstrap 與心跳設計

Bootstrap 問題：第一個憑證怎麼來？agent 要向管理面證明「我是合法的新叢集」。標準模式（kubelet TLS bootstrapping 同款）：管理員產生一次性註冊 token（短命、綁定叢集名）→ agent 用 token 首次連線 → 管理面驗證後簽發正式的 client 憑證（綁定 cluster identity）→ 之後全部用憑證，token 作廢。本週實作可簡化為「預共享 token + 首次連線換發長期 token」，但要能講出完整版。

心跳的內容與頻率：Report 每 30s 帶上版本/節點數/容量/agent 版本；管理面收到後更新 Cluster status 的 LastHeartbeatTime。頻率的權衡：太密=管理面負載隨叢集數線性增長；太疏=故障偵測延遲。進階做法是分層——heartbeat 輕量高頻（只報「我活著」），full report 低頻或變化時才報（capacity 變了、版本升了）。

任務本週實作清單

  • agent.proto + buf generate；管理面加 AgentService 實作（in-memory 的 per-cluster 指令佇列與 seq 管理）。
  • cluster-agent 部署進 member-b（Deployment + 最小 SA权限），Direct 模式的探測邏輯對 Agent 模式叢集切換為「看心跳」。
  • 斷線演習：殺管理面 → agent 退避重連（觀察 jitter）；殺 agent → 管理面佇列堆積 → agent 回來全部補送且不重複執行。
  • 重複投遞演習：管理面故意重發同一 instruction，驗證 agent 去重。

✅ 第 26 週驗收清單

  • agent 上線後 Cluster status 由心跳驅動更新；斷心跳 90s 後 phase=Unknown。
  • 斷線期間下發的指令，重連後按序執行、結果回報齊全、無重複執行。
  • agent Pod 重啟後 seq 續傳正確（持久化生效）。
  • 能畫出完整的訊息時序圖（含斷線窗口）——這張圖就是 Phase 8 產出的一部分。

Q1reverse connection 解決了哪三個問題？

顯示答案

(1) 網路可達性：member 在 NAT/防火牆後，管理面連不進去，但 member 可以出站——連線反向後只需單向的出站 443。(2) 憑證集中風險：管理面不再持有全部叢集的 kubeconfig；agent 用叢集本地 SA 執行操作，憑證不出叢集。(3) 延展性：每叢集一條 stream 由 agent 維護，管理面不用對 N 個 API server 各開 watch。

Q2sequence number 和 ack 各解決什麼？只有其中一個會怎樣？

顯示答案

seq 解決「斷線續傳與去重」（收方知道處理到哪、跳過重複）；ack 解決「送方何時能忘記」（收到結果才算送達，否則重發）。只有 seq 沒 ack：送方不知道對方收到沒，斷線時只能盲目重送或丟失。只有 ack 沒 seq：重發的訊息收方無法識別重複，非冪等操作被執行兩次。兩者合成 at-least-once + 有序去重。

Q3「先執行後記 seq」vs「先記 seq 後執行」的語義差異？你選哪個、配什麼？

顯示答案

先記後執行：crash 發生在記錄與執行之間 → 該指令永遠不會重做 → at-most-once（可能丟活）。先執行後記：crash 在執行與記錄之間 → 重連後該指令重做 → at-least-once（可能重複）。選 at-least-once + 讓執行冪等（SSA apply 天然冪等）——丟活是靜默錯誤，重複是可被冪等吸收的噪音。與 workqueue 的 Done 語義同構。

目標：補完多叢集三部曲的最後兩塊：把 desired state 可靠地種到 member 叢集（SSA + drift 修復），以及「工作負載該去哪個叢集」的 placement 決策。做完這週，你的平台就是一個小號的 Karmada。

27.1Remote apply：SSA 與 field manager 的主場

把 manifest 種進 member 叢集，正確工具是 W11 埋下的 Server-Side Apply：

SSA 在這個場景的三重價值：冪等（重放同一 manifest 無害——W26 的 at-least-once 靠它兜底）；field manager 隔離（member 叢集本地的 HPA 改 replicas、有人加 annotation，都不會被我們的 apply 蓋掉——只有「platform-agent 管理的欄位」被強制對齊）；drift detection 免費——下面說。

Drift 的處理哲學：有人在 member 叢集手改了平台管理的資源怎麼辦？兩個機制疊加：(1) 週期性重新 apply（agent 對已下發的 desired state 每 N 分鐘重放一次 SSA）——「平台管的欄位」自動修復漂移；(2) 平台不管的欄位放任自由——這是刻意的邊界，全面鎖死請用 admission policy，不要用無限對抗的 apply 迴圈。

27.2Failure handling：部分失敗是常態

多叢集操作的本性：對 N 個叢集做一件事，結果永遠是「K 成功、M 失敗、其餘未知」。設計原則：

  • 結果建模為 per-cluster status：Workload CR 的 status 是一張表（cluster → phase/message/lastApplied），不是單一布林。使用者要能看到「A 好了、B 失敗因為 quota、C Unknown」。
  • Cluster unavailable ≠ 失敗：目標叢集 Unknown 時，指令留在佇列（W26），status 標 Pending——恢復後自動送達。只有明確的拒絕（quota 超、RBAC 拒、validation 失敗）才是 Failed。
  • Timeout 與 retry budget：每叢集的操作有獨立 timeout；重試消耗 per-cluster 的預算（如 5 次/10 分鐘），耗盡標 Failed 並停止——不因一個壞叢集無限燒 CPU。healthy 叢集的操作絕不等待 unhealthy 叢集（並行、隔離，W25 的 QPS/timeout 在此閉環）。

目標：進入 AI infrastructure 的核心地帶。先想清楚「serve 一個模型」到底需要哪些零件，再看 KServe 如何把這些零件標準化成一個 CRD——以及它替你省掉的東西正是你已經會自己做的東西。

28.1Serve 一個模型需要什麼：先手工再框架

28.2KServe InferenceService：宣告式的模型部署

KServe 收走的活：storage initializer（init container 拉模型——你上週手寫的那個）、runtime 選擇（按 modelFormat 配對 ServingRuntime）、autoscaling（含 scale-to-zero）、路由與 status 回報。三個角色概念：predictor（跑模型的主體）、transformer（前後處理：特徵轉換、tokenize）、explainer（可解釋性）——組成一條推論管線，多數場景只用 predictor。

Q1模型服務的 readiness 和一般 API 服務有什麼本質不同？

顯示答案

顯示答案

GPU Scheduling、LLM Runtime 與 Inference Routing

目標：AI infra 的三個縱深主題：GPU 怎麼被 K8s 排程與切分、LLM serving 為什麼是全新的效能問題（KV cache、continuous batching）、以及推論流量需要什麼樣的特殊路由。沒有 GPU 也能完成 control plane 側的全部實作。

30.1GPU 在 K8s 裡怎麼運作：device plugin 到排程

K8s 原生只認 CPU/memory，GPU 靠 device plugin 機制接入：NVIDIA device plugin（DaemonSet）向 kubelet 註冊 nvidia.com/gpu 這個 extended resource，node 的 allocatable 出現 GPU 數量，Pod 用 resource request 索取：

三個 GPU 特有的現實：

  • GPU 不可超賣、預設不可共享：nvidia.com/gpu: 1 是獨占一整張卡。小模型佔大卡＝浪費，這就是共享技術存在的原因：MIG（A100/H100 等的硬體切分：一張卡切成數個隔離實例，各自成為獨立的 extended resource，隔離最強）與 time-slicing（軟體分時，無記憶體隔離——多租戶慎用）。
  • GPU fragmentation：需要 4 GPU 的工作排不進「兩台各剩 2 張」的叢集——碎片化。對策：bin-packing 傾向的排程（W27 的 score 反向）、大工作優先、gang scheduling（要嘛全排要嘛不排，訓練場景）。
  • 可觀測：DCGM exporter 出 GPU utilization/memory metrics——「GPU 排滿了」≠「GPU 用滿了」，利用率是成本優化的核心指標。

無 GPU 環境的替代練習（本週的實作路徑）：給 kind node 打上假 extended resource（對 node status 的 capacity patch example.com/mock-gpu: 4）+ label + taint——排程語義完全一樣，placement、fragmentation 實驗照做。

30.2LLM serving：為什麼是全新的問題

LLM 推論的兩個階段決定一切：prefill（處理輸入 prompt，算力密集、可並行）→ decode（逐 token 生成，每個 token 都要讀全部歷史的 attention 狀態——記憶體頻寬密集）。歷史狀態就是 KV cache：每個進行中的請求都占一塊與「序列長度 × 模型層數」成正比的 GPU 記憶體。推論的併發上限常常不是算力而是 KV cache 的記憶體。

由此催生的 runtime 技術（vLLM 的成名作）：

  • PagedAttention：KV cache 分頁管理（作業系統虛擬記憶體的思路），消除長短請求混跑時的記憶體碎片。
  • Continuous batching：靜態 batch 要等整批全部生成完；continuous batching 在每個 token 步動態進出請求——先完成的先走、新請求隨時插入，吞吐提升數倍。這是「LLM 專用 runtime」存在的核心理由。

Runtime 定位速覽：vLLM（LLM 專用、吞吐王者、OpenAI 相容 API）；TGI（HuggingFace 生態）；Triton（NVIDIA 的多框架通用 server，LLM 場景配 TensorRT-LLM backend）。Model parallelism 概念：模型大過單卡記憶體時的切法——tensor parallel（層內切，卡間高頻通訊，同機多卡）與 pipeline parallel（按層切，跨機可行）——平台側你需要懂的是它決定了「一個副本=幾張卡、什麼拓撲」，直接影響排程與擴縮的粒度。

Batch、latency、throughput 的三角：batch 越大吞吐越高（GPU 利用率），但每個請求的等待越久。LLM 還多一個維度：TTFT（time to first token，交互體驗）vs 總生成時間——聊天場景優化 TTFT + streaming，批次場景優化吞吐。平台要允許 per-model 的這類調參（max batch size、max concurrent），這是 ModelService spec 未來的演進方向。

30.3Inference routing：為什麼普通 LB 不夠

W22 的 weighted routing 處理「請求都一樣」的世界；推論流量打破這個假設：

  • 請求成本高度不均：8 token 和 8000 token 的請求成本差千倍——round-robin 會把長請求堆在同一副本。
  • 副本有狀態：KV cache 佔用不同、佇列深度不同——最優路由要看副本的即時負載（queue depth、cache 餘量），而不是連線數。
  • 語義路由需求：同一端點後面多個模型版本/LoRA adapter，按請求內容（model 欄位、header）選 backend；prefix cache 親和（同對話路由到同副本以復用 KV cache）。

Gateway API Inference Extension 就是為此而生的標準化嘗試：InferencePool/InferenceModel 資源 + endpoint picker 機制，讓 gateway 的路由決策能諮詢「懂推論」的排隊/負載邏輯。定位要講得準：它是 Gateway API 的擴展（W22 的知識直接續用），目標是把「LLM-aware 路由」從各家私有實作變成標準介面。

Canary model version 的推論特色：除了流量比例（W22/23 的 90/10），模型 canary 還要比較品質指標（回應長度分佈、拒答率、下游評分）而不只是錯誤率與延遲——「新版本 200 OK 但答案變爛」是 HTTP 指標看不見的。實作上：路由層打 version 標籤 → 推論 log/metrics 按版本切分 → 評估管線比對。

Q1extended resource 和 CPU/memory 的排程語義差在哪？GPU 為什麼 request 必須等於 limit？

顯示答案

extended resource 是整數且不可壓縮：不能像 CPU 那樣 throttle 共享，也沒有 memory 那樣的 OOM 回收語義——一張卡就是被一個容器獨占（除非 MIG/time-slicing 顯式切分）。因此不允許 request≠limit 的超賣（沒有機制在爭用時仲裁）。排程上它只是 node allocatable 的計數；隔離由 device plugin 掛載對應設備完成。

Q2KV cache 是什麼？為什麼它決定 LLM 服務的併發上限？

顯示答案

decode 階段每生成一個 token 都要對全部歷史 token 做 attention；KV cache 把歷史的 key/value 張量存在 GPU 記憶體避免重算，大小 ∝ 序列長度×層數×batch。模型權重佔掉固定記憶體後，剩餘空間能容納的 KV cache 決定同時在飛的請求數（及其長度）——算力還有餘但記憶體滿了就得排隊。這也是 PagedAttention（分頁消碎片）與 prefix caching（共享前綴復用）的價值所在。

Q3continuous batching 相對靜態 batching 改進了什麼？

顯示答案

靜態：湊一批一起跑，整批要等最長的請求生成完，短請求陪跑、新請求等下一批——GPU 在批尾大量空轉。continuous：排程粒度降到單個 token 步，每步都能移出已完成的、插入新到的請求，GPU 始終滿載。效果是吞吐數倍提升且平均等待下降——代價是 runtime 複雜度，這正是 vLLM 這類專用引擎的護城河。

Q4推論流量的路由為什麼不能用普通的 round-robin/least-conn？

顯示答案

假設破裂：請求成本不均（token 長度差千倍，連線數不代表負載）、副本有狀態（KV cache 餘量、佇列深度各不同）、還有語義維度（模型版本/adapter 選擇、對話親和以復用 cache）。正解是 load-aware + semantic routing：路由器諮詢副本的即時指標（queue depth、cache 佔用）與請求內容做決策——Gateway API Inference Extension 就是把這個「諮詢機制」標準化的嘗試。

面試問答 · Week 30

為 LLM 推論平台設計 autoscaling，訊號和困難點？

訊號：RPS 幾乎無用（成本不均），用併發請求數、queue 深度、KV cache 佔用率、TTFT p95——vLLM 這些都有 metrics。困難點：(1) 擴容延遲巨大（拉大 image + 載入數十 GB 權重，分鐘級）——要預測性擴容或保暖池，純反應式必然雪崩；(2) 縮容要 drain（進行中的長生成不能殺）——preStop + 停止接新請求；(3) 副本粒度粗（tensor parallel 的一個副本=N 張卡）；(4) scale-to-zero 對大模型的喚醒成本可能高於常駐。結論：LLM 的 HPA 是「訊號選對 + 提前量 + 優雅排水」的系統工程，不是套個 CPU 閾值。

GPU 利用率很低但排程已滿，怎麼辦？

先區分兩種「滿」：分配滿（allocatable 用完）vs 利用滿（DCGM 的 SM/memory 利用率）。分配滿+利用低=典型的粒度錯配：小模型獨占大卡。對策階梯：MIG 把大卡切成小實例（硬隔離，多租戶安全）→ time-slicing（軟共享，信任邊界內）→ 模型合併部署（一個 runtime 服務多模型，Triton 的強項）→ bin-packing 排程減少碎片 → 建立 GPU 利用率的 showback/chargeback 讓租戶有動機選對規格。順序反映隔離與複雜度的權衡。

Q1contract test 和 buf breaking 各防什麼？為什麼兩個都要？

顯示答案

buf breaking 防「proto 定義層」的破壞（field number/型別變更）；contract/golden test 防「行為層」的破壞——proto 沒變但序列化輸出變了（欄位語義改動、mapping 程式碼重構出錯、預設值行為變化）。前者看契約文件，後者看實際回應；client 感受到的是後者，工具能自動擋的是前者，缺一不可。

Q2failure injection 演練的三個評分問題是什麼？為什麼「系統沒掛」不算過關？

顯示答案

(1) 爆炸半徑受控嗎（故障是否被隔離在其邊界內）；(2) 使用者可自助理解嗎（status/conditions 足以說明，不需平台工程師看 log）；(3) 恢復是否自動（修根因後系統自行收斂）。「沒掛」可能只是運氣或掩蓋——例如錯誤被吞掉、狀態停在舊值，表面平靜實則失真，這比明確的失敗更危險。

Q3govulncheck 相對一般相依掃描工具聰明在哪？

顯示答案

一般 SCA 只看 go.mod 裡的版本號比對 CVE 清單——你引了有漏洞的套件就報，噪音大。govulncheck 做符號級可達性分析：只有當你的程式碼（經呼叫圖）真的會執行到漏洞函式才告警。結果是低噪音、高可信的報告，修復優先級清楚——這也讓「零 HIGH 容忍」的 CI 政策實際可行。

三線並行：(1) 公開與傳播——repo 公開、每個 Phase 寫一篇技術文（「事件不等於狀態」那篇最有傳播力）、在社群分享 demo；(2) 補真實性——真 GPU 環境跑一次 vLLM 路徑、把 known limitations 裡挑一條解掉（管理面 HA 是好選擇：leader election 你已經會了）；(3) 對照業界——精讀 Karmada/OCM/KServe 的原始碼各一個模組，寫「我的實作 vs 它們的實作」比較筆記——面試時這是「有真實觀點」的最硬證明。
